1. 请基于我的笔记总结 Prompt AB 测试的标准五步法。：
根据你的笔记，Prompt AB 测试的标准五步法可以总结为下面这五个步骤，整体逻辑很清晰，核心就是把“优化 Prompt”这种模糊的事，变成可量化、可复现的数据实验：

第一步：明确实验目标，锁定单一变量
先定一个具体要解决的问题（比如提升对话轮次、降低投诉率），然后只改一个 Prompt 元素——像“共情引导 vs 直接建议”这样，其余语气、知识库等全部保持不变。一次只动一个变量，才能知道效果到底是哪个改动带来的。

第二步：分层定位，保证样本有效
用户特征差异会影响效果，得按新老用户、年龄、使用场景（比如深夜emo vs 日常闲聊）来分层。每组至少需要1000个独立用户或2000条有效对话，并且流量要随机分配，避免把活跃用户全都分到一组去。

第三步：设计核心指标与辅助指标
指标必须贴合用户感知和业务目标。比如想提升对话粘性，核心就看对话轮次和7日留存；想把人设做稳定，核心就看投诉率和人工抽检匹配度。辅助指标可以补充观察，比如主动对话比例、关键词匹配等。避免用“模型准确率”这种技术指标，因为用户感知不直接关联。

第四步：执行中监控异常与显著性
实验期间要防止变量泄露——用户如果公开对比两组 Prompt 会导致反馈失真，所以不能在公开渠道讨论。同时建一个实时看板，一旦某组突然大量投诉就暂停排查。当差异达到95%置信度时（p < 0.05），可以考虑提前结束；如果显著性没出来就延长周期。

第五步：解读结果并沉淀记录
看完胜负之后，必须分析“为什么好”——用统计工具验证显著性，确认是变量导致的差异。然后把优胜版本全量上线，同时把实验过程、关键结论都记到 Prompt 设计文档或迭代历史里，方便后续复用，避免重复劳动。

这套方法已经很成熟了，你笔记里各份材料高度一致，没有冲突。所以实际执行时，按照上面五步走就行。要提醒一点：样本量达标和一次只改一个变量是成功的基础，如果这两点没做到，后面解读可能不靠谱。另外，实验周期也要留够，不要因为看到短期波动就急着下结论——95%置信度这个门槛不能降，否则容易把随机波动当成真效果。

2. 如果我要提升对话留存，AB 测试里核心指标和辅助指标应该怎么设？
如果要通过 AB 测试来提升对话留存，最稳妥的指标设置方案是：核心指标选“7日留存”和“对话轮次”，辅助指标加“主动发起对话比例”和“用户满意度”。

这个做法的依据很简单：留存是业务目标最直接的体现——7日留存能看出用户是否愿意长期回来，对话轮次则能反映单次体验是否足够有吸引力，这两个指标绑在一起，既看长期粘性也看单次深度。辅助指标里，主动发起对话比例可以帮你判断用户是不是被你的引导方式激发了主动交流意愿，而用户满意度（比如直接打分或投诉率）能兜底，避免为了拉留存牺牲体验——比如强行延长对话却让用户反感。至于那些“模型准确率”之类技术指标，跟用户感知脱节，最好别作为留存实验的主参考。

当然，这套指标要生效，前提是实验设计必须规范：一次只改一个 Prompt 变量（比如回复语气或引导方式），并且按新老用户、使用场景分层分配流量，保证每组样本量足够（通常每组至少1000个独立用户或2000条有效对话）。如果你的实验场景比较特殊，比如产品本身用户基数小或者对话内容偏工具型，那7日留存的波动可能会比较大，这时候可以适当延长观察周期，或者把“次日留存”作为临时核心指标来减少噪音。

3. 这个和前一个问题相比，最关键的实验控制变量是什么？
最关键的实验控制变量始终是**“一次只改动一个 Prompt 元素”**，其余部分保持完全一致。无论是五步法还是留存指标实验，都得先把这一点卡死，否则后续结果根本没法归因——你不知道到底是语气变了、知识库不同了，还是别的因素造成的差异。

这个原则在标准五步法里是第一条要求，也是后续所有分层、指标设计、监控和解读的根基。比如五步法第一步就会明确说“锁定单一变量”，示例里也专门用了“共情后引导提问 vs 直接给建议”这种只改一个点的对比。等你具体到提升留存的实验时，要改的可能就是回复时的共情语气或者引导方式，但依然只动这一个点，语气、知识库、对话结构都不变。只有这样做，留存指标的波动才能确定是那个变量带来的，而不是其他因素混进来干扰。

有一点值得注意：虽然“单一变量”是通用的硬性条件，但具体到不同实验目标，你锁定的那个变量本身会不同。比如五步法里可能改的是建议方式，而留存实验里改的可能是回答的亲切程度或者信息密度——变量内容不一样，但控制逻辑完全一样。所以不用纠结“这个实验和上个实验哪个变量更重要”，关键是实验设计时有没有严格遵守“一次一动”的规则，这个规则本身才是所有实验的最关键的基石。

4. 换个话题：AI产品经理的 PRD 模板应该包含哪些模块？

好问题。AI产品经理写PRD，跟传统互联网产品的PRD确实是两码事，核心就在于多了几个AI特有的交付物需要你去定义和论证。

一个最稳妥、能直接用起来的模板，至少应该包含下面这几个核心模块：

1. 背景与价值论证（为什么非用AI不可？）
这是AI产品PRD的开篇灵魂，不能只写“用户想要个功能”。你得说明白：为什么传统方案搞不定（比如规则写不完、成本过高），大模型在这个场景里到底不可替代在哪里，最后再算一笔账——投入的算力、人力换来的收益值不值。这个论证过不了，后面的方案根基就不稳。

2. 系统流程图（LLM视角的5类节点）
跟传统流程图不一样，你得专门标出5类特殊节点：LLM调用节点（模型对话）、工具调用节点（比如查天气API）、判断/路由节点（if-else逻辑）、人工介入节点（兜底给真人），以及异常处理分支（模型崩了怎么办）。这样开发一看就知道哪些地方要接模型，哪些要走规则。

3. 模型选型与约束（红线 vs 弹性线）
这一步是技术决策的核心。你得给出两条线：红线（Must-have，比如必须能商用、必须支持高并发、响应时间不能超过3秒）和弹性线（Nice-to-have，比如参数量小、推理成本低）。记住，任何一项红线不达标，这个方案就一票否决。另外，尽量优先用小参数模型省成本。

4. System Prompt设计与版本管理
Prompt不再是写文案，而是产品文档的一部分。推荐用模块化写法，比如用 # ROLE、# BOUNDARY、# WORKFLOW 这样的Markdown标签把角色、边界、指令拆开。变量统一用双大括号 {{ }} 占位，方便迭代。Prompt的版本要按语义化版本号（比如1.2.0）管理，每次改动都必须跑评测及格才能上线。

5. 训练数据集（Q&A对 + Chat格式）
不能直接丢给开发一份PDF让他们去微调。必须由运营或标注团队，把它转化成“模拟用户提问 -> 专家标准回答”的 Q&A对，或者完整的多轮对话 Chat格式。这是模型学习的原料，格式不规范，微调出来就是一团乱码。

6. 评测体系（离线跑分 + 灰度监测）
必须搞一套专门的评测集，来源包括：种子用户提供的bad case、生产日志里的点踩记录、业务专家找的边界场景、甚至是“对抗靶场”（故意刁难模型）。存储成标准的ChatML JSON格式，嵌入到CI/CD流水线里——每次改动Prompt或模型版本，自动跑分，评分低于旧版就直接打回。

7. 灰度发布与迭代节奏
先放5%10%的流量，观察2448小时，确认没有系统故障或大面积投诉，再逐步推到50%再到全量。迭代周期建议：紧急Bug修复24小时间闭环，常规功能迭代按双周Sprint跑。每次上线都要回滚预案。

8. 成本控制与质量控制
成本控制要警惕一个陷阱：模型的输出Token通常比输入Token贵3~4倍，所以写Prompt时得想怎么缩短模型回复长度。隐藏成本还包括显卡折旧、机房电费、运维人力。质量控制则要构建纵深防御：输入层过滤有害词，推理层控制输出格式，输出层加拦截逻辑（比如模型乱说人话时直接截断并返回兜底回复）。同时必须在界面上声明产品能力边界，超出范围直接提示“我干不了这个”。

最后说两点你可能容易忽略的边界：第一，这个模板不是填表式的文档，它更偏向一份工程级的技术方案，每一块都需要你带入实际业务数据去填充。第二，特别重视版本规范和成本陷阱，这两样很容易在项目后期反噬，导致返工或超预算。如果你手头正有一个AI项目在构思，可以从“背景论证”和“模型选型红线”这两块先搭建骨架。

5.继续上一个 PRD 话题，给我一个可直接套用的最小模板。
好的，这就给你一个能直接套用的最小 PRD 模板，基于 Agent 的“感知→决策→执行→反馈→学习”闭环来设计。你照着这几个模块填内容就行，不用纠结格式，重点是逻辑闭环。

模板结构（五步闭环）：

背景与价值
写清楚：为什么非要用 AI/Agent 来解决？传统手段为什么搞不定？用一句话说破核心收益（比如“将客服一次解决率从 40% 提升到 70%”）。

感知与输入
定义 Agent 能接收哪些信息：用户提问、上下文、外部信号（比如时间、地点、用户画像）。列出关键数据字段和触发条件。

决策与路由
说明 Agent 怎么判断该做什么：是直接调用模型回答，还是走规则分流，还是需要调外部工具（比如查知识库、查 API）。画出最简流程——if-else 节点和 LLM 调用节点。

执行与输出
定义 Agent 输出什么：是文字回复、结构化数据还是触发某个动作。直接贴 System Prompt 的骨架（用 # ROLE 和 # WORKFLOW 标签写明角色和步骤），并用 {{变量}} 标出动态填充位置。

反馈与学习
写明怎么收集用户反馈（点赞/点踩、投诉率、断流率），以及这些数据如何回流到下一轮迭代——比如每周跑一次 Bad Case 分析，更新评测集，然后修改 Prompt 或策略。

这个模板背后的逻辑很简单：AI 产品本质上是一个闭环系统，而不是堆砌功能的静态页面。你把输入、决策、输出、反馈、迭代这五个环节写清楚，开发和测试就能知道每个环节该做什么。而且它不依赖任何特定技术框架，不管你是用 GPT 还是本地小模型，都适用。

需要注意的边界：
如果你们项目还在非常早期的 POC 阶段（比如连具体场景都没定），可以直接跳过“反馈与学习”，先把前四步写出来，后面再补上。另外，如果你的 Agent 需要集成多个外部工具（比如同时调天气 API 和用户数据库），那“感知与输入”就要详细列出所有外部数据源的接入方式。这个最小模板帮你省掉的是排版和废话，但核心数据和技术细节不能省——否则写出去了也没法落地。

6.再换话题：意图识别准确率评估要看哪些指标，为什么？ 
评估意图识别准确率，最稳妥的指标组合是：精准率、召回率、F1 值，同时结合场景调整优先级，并配套动态监控与业务验证。

原因很简单：精准率告诉你“我判定的意图里，有多少是真正对的”；召回率告诉你“所有真实意图里，我找出来了多少”；而 F1 值把两者综合起来，避免只盯一边导致偏科。比如在智能客服投诉场景里，漏掉一个投诉意图可能引发用户爆发，所以召回率优先，宁可误判几个，也要把投诉全抓住；但在电商导购场景里，推荐错了商品用户扭头就走，这时精准率优先，宁可少推荐，也不能乱推荐。

光看这几个数字还不够——你得保证数据能反映真实情况。比如评估数据集至少要 90% 来自线上真实用户 query，覆盖常见意图、边缘意图、甚至乱码等低质输入；标注标准要由算法和运营一起定，多人交叉标注一致性达到 90% 以上；然后定期做误差分析，分清楚是意图混淆、语义偏差还是罕见意图没覆盖。同时，多轮对话场景里还要额外评估模型能不能结合上下文修正意图，这个可以通过专门的多轮样本准确率来测。

最后，以上所有指标都是过程指标，终极标准是业务结果。如果精准率、召回率都挺高，但客服进线率没降、用户满意度没升，那说明要么意图分类体系本身有问题，要么识别出的意图没被正确执行。这时候就得回头重新校准意图定义，而不是单纯在模型参数上使劲。

需要注意的是，这套评估体系不是一次性搞定的，必须跑在线上实时监控 + 每周线下迭代的闭环里——一旦发现准确率波动或用户频繁点踩，马上补充样本、优化分类逻辑、调整上下文关联策略。这样，意图识别才能真正为业务落地保驾护航。
