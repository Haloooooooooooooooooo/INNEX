Prompt AB测试五步法
核心内容
Prompt AB测试的核心理念与传统产品AB测试一致，但将测试变量从功能/UI改为文本规则。
整个流程分为五步：明确目标→分层人群→构建指标→执行监控→结果解读与落地。
关键目标是将模糊的“优化Prompt”转化为可量化、可复现的数据结论。
每一步都强调避免干扰、保证样本有效性与统计显著性（如95%置信度）。
---

关键概念 / 关键信息
单一变量原则：测试时只改一个核心维度（如回复逻辑），其他因素（语气、知识库调用）必须固定。
分层定位：按新老用户、年龄、性别、使用场景（如深夜emo vs 日常闲聊）分层，避免偏差。
样本量要求：每单组至少1000独立用户或2000有效对话。
评估体系：核心指标与业务目标绑定（如对话轮次、7日留存）；辅助指标如主动发起对话比例。
统计显著性：使用置信工具验证，差异需达到95%以上才认为是实验变量导致。
风险监控：变量泄露（用户对比截图）、异常数据波动（大量投诉）需及时排查。

3. `/compact` 将对话压缩成摘要，释放上下文空间；Claude Code 在上下文接近 95% 时会自动触发压缩。  
4. `/resume` 可恢复之前的会话，默认显示当前工作空间的会话。  
5. `/branch` 和 `/fork` 作用相同，从当前对话分支出新会话，原对话保持不变。  
6. `/rewind` 可回退到任意检查点（每次按回车执行前的状态），同时回退对话记录和文件改动。  
7. `/btw` 用于快速插入不相关的小问题，不会污染当前对话上下文。  
8. `/usage` 显示 token 用量和费用估算；若通过 API Key 接入第三方模型，费用估算可能不准确。  
9. `/context` 可视化上下文窗口使用情况，展示占用因素是对话记录、文件还是系统提示词。  
10. `/plan` 进入规划模式，AI 先通过交互提问确认信息，再输出详细实施方案，待确认后才开始写代码。  
11. `/diff` 展示所有未提交的代码变更，可用左右键切换整体 diff 和每轮改动。  
12. `/status` 可在 Claude 当前回复未结束时随时调用，显示版本号、模型、账户信息等。  
13. `/insights` 生成使用分析报告（默认英文），可投喂给 AI 进一步分析。  
14. 原文提到“日常使用时，直接用中文跟 Claude 说你的需求……它会自动帮你执行对应的命令”。  
15. 原文建议“当 `/context` 显示上下文用量超过 80% 时就执行一次 compact”。

## 推断与延展（非事实）

1. 了解这些命令有助于用户更精确地控制 AI 交互行为，避免纯聊天模式下可能出现的上下文污染或费用浪费。  
2. 主动管理上下文（如每完成一个子任务执行 `/compact`）比依赖自动压缩更可靠，能减少关键细节被压缩的风险（依据：事实 3、15）。  
3. `/plan` 模式适合需要严格分步执行的任务，可降低 AI 偏离轨道或输出无效代码的概率（依据：事实 10）。  
4. `/btw` 的设计暗示纯聊天模式下临时问题的加入会持续占用上下文，而 `/btw` 提供了一种“用完即弃”的轻量交互方式（依据：事实 7）。

6. `/insights` 报告可帮助用户发现自己的 AI 使用习惯和瓶颈，从而优化工作流（依据：事实 13）。

## 核心内容

1. 意图识别是Agent理解用户自然语言并转化为可执行指令的关键环节，包含定义、流程和原则三方面。
2. 核心流程分为四步：预处理、意图分类、槽位抽取、结果合并，最终交付给Agent执行。
3. 三个关键原则（单一职责、可执行性、无歧义性）确保意图定义清晰、无重叠、可映射到具体工具。
4. 文章定位是面向产品经理的入门级介绍，鼓励有兴趣者进一步深入学习。

## 关键概念 / 关键信息

- **意图识别**：将用户自然语言映射到预定义意图标签并抽取关键参数的过程。

- **槽位抽取**：提取执行意图所需的参数（如时间、地点、数量等）。

- **三个原则**：
  - 单一职责：一个意图只对应一个核心动作。
  - 可执行性：意图必须能映射到具体函数/工具。

# 意图识别准确率评估四环节

## 核心内容

1. 意图识别准确率评估需从定义边界、标准化流程、场景差异化、动态迭代四个环节展开。  
2. 核心指标包括准确率（精准率）、召回率、F1值，业务场景决定指标优先级。  
3. 评估流程需构建高质量数据集（90%+真实用户数据）、统一标注规范（标注一致性≥90%）、执行测试与误差分析。  
4. 不同场景侧重点不同：投诉场景优先召回率，导购场景优先准确率，且需评估多轮对话的上下文修正能力。  
5. 建立线上实时监控与线下周更新迭代闭环，通过补充样本、优化分类、调整参数等手段持续提升。  
6. 终极验证标准是业务结果（如满意度、转化率），而非单纯准确率数值。

## 关键概念 / 关键信息

- **意图识别准确率**：模型预测意图与真实意图的匹配程度。  
- **评估范围**：需区分意图类别边界，排除无意义乱码、不完整语句等无效样本。  
- **核心指标**：准确率（精准率）、召回率、F1值（两者调和平均）。  
- **评估数据集**：覆盖常见、边缘、相似、低质四类输入；线上真实用户query占比≥90%。  
- **标注规范**：算法与运营团队共同制定，多人交叉标注确保一致性≥90%。  
- **误差类型**：意图混淆、语义理解偏差、罕见意图未覆盖。  
- **场景化要求**：  
  - 投诉意图：召回率优先（漏判严重影响体验）。  
  - 电商导购：准确率优先（推荐错误导致用户流失）。  
- **线上监控**：实时看板跟踪核心意图准确率，设置预警阈值，结合用户反馈。  
- **线下迭代**：每周更新评估集重测，根据误差结果补充训练样本、优化分类体系、调整模型参数、增加上下文关联。  
- **终极标准**：业务指标，如客服进线率下降、满意度提升、导购转化率上涨。

## 原文支持要点（事实）

1. 准确率的本质是“模型预测的意图与真实意图的匹配程度”。  
2. 评估前需“界定评估范围”：区分意图边界（如查询订单 vs 咨询退款），排除无效样本（如乱码、不完整语句）。

## 推断与延展（非事实）

1. 原文强调“90%以上用户数据”贴近真实场景，暗示线上数据与线下模拟数据可能存在分布偏差，需持续从线上采集反馈以保持评估有效性。

- **PM懂技术的保留红线**：不用写代码、调试程序、搭服务器；必须懂基本原理、看懂架构图、理解数据流、知道方案优缺点。
- **客户端-服务端模式**：用户通过浏览器（客户端）发起请求，服务器处理后返回结果，是Web最基础的交互逻辑。
- **HTTP请求方法**：GET（获取）、POST（提交）、PUT/PATCH（修改）、DELETE（删除）。
- **关键状态码**：200（成功）、401（未授权）、404（找不到）、500（服务端错误）。
- **登录态三种方式**：Session（服务端记录）、Token/JWT（客户端携带）、Cookie（客户端存储小文件），其中Token是当前最常用的方式。
- **前后端分离**：前端负责页面展示和交互，后端负责业务逻辑和数据处理，通过API（接口）沟通。
- **前端三件套**：HTML（骨架）、CSS（样式与布局）、JavaScript（交互与逻辑）。
- **最常用前端框架**：React（适合大型复杂产品）、Vue（国内常用，适合中小型产品/小程序）。
- **组件化**：将页面拆解成可复用的“小零件”（组件），是前端开发高效的关键思想。
- **状态管理**：页面所有变化本质上是“状态”的变化，分为全局状态和局部状态。

2. HTTP与HTTPS的核心区别是HTTPS多了加密，更安全，适用于传输敏感信息（如密码、支付信息）。
3. 状态码200代表成功，401代表未授权，404代表找不到，500代表服务端错误。
4. 登录态的三种方式是Session（服务端记录）、Token/JWT（客户端携带）、Cookie（客户端存储小文件）。

6. 组件化的好处：重复使用减少开发时间，界面规范易于维护，适配复杂产品。
