## 2026-06-01｜项目定位与竞品差异（第一轮）

### Q1：这个项目最初想解决的到底是什么问题？

### 结论

这个项目最初不是从“做一个 AI 功能很多的知识工具”出发，而是从两个连续痛点出发：

1. **收藏即遗忘**
- 用户在多个平台收藏了很多内容，但信息分散、缺少统一入口，久而久之既忘了看，也忘了复习。
- 所以系统第一层价值，是把分散碎片内容统一收进一个待处理池，降低“收了但不处理”的损耗。

2. **看过不等于内化**
- 很多内容即使后来重新打开，也只是“看过”，没有真正转化成自己可复用的知识。
- 后续如果再想找、想讲、想用于判断，还要重新理解一次，时间成本很高。
- 所以系统第二层价值，是通过内化、结构化笔记、图谱和问答，把内容变成可复用的知识资产。

### 当前项目更适合的产品定义

相比“AI 知识管理工具”这种泛化说法，这个项目更适合定义为：

**一个把分散碎片内容低门槛沉淀为可复用知识资产的 AI 知识系统。**

其中“知识资产”比“知识管理”更准确，因为项目重点不只是存储，而是后续可调用、可复用、可追溯。

---

### Q2：它和 Obsidian、NotebookLM 这类成熟产品相比，差异在哪里？

### 结论

这个项目不适合和竞品做“功能有无”的简单比较，而更适合从**目标任务、默认工作流、产品心智**三个层面对比。

#### 1. 与 Obsidian 的差异

Obsidian 的核心优势是：
- 本地优先
- 开放格式
- 插件生态强
- 高自由度、可塑性强

它更像一个**强大的知识工作台**，适合重度用户长期搭建自己的系统。

但其代价是：
- 启动成本高
- 需要自己决定目录、标签、模板、插件组合
- 后续还需要持续维护工作流

因此，不是 Obsidian 不好，而是它解决的是“如何拥有一套强大的知识工作台”。

本项目切的不是这个方向，而是：
- 降低启动和维护成本
- 提供默认就能跑通的收录 -> 待处理 -> 内化 -> 问答闭环
- 服务那些“有知识处理需求，但不想自己搭系统”的用户

一句话概括：

**Obsidian 解决的是自由度问题，而本项目更想解决的是低门槛闭环问题。**

#### 2. 与 NotebookLM 的差异

NotebookLM 的核心优势是：
- 基于 source 的理解和问答能力强
- 多源资料处理能力强
- 更适合围绕一批材料做快速研究、总结和问答

它更像一个**智能研究助手**。

本项目关注的则是：
- 长期分散收藏的统一收录
- 内容的持续内化
- 跨笔记关系形成
- 长周期知识复用，而不是一次性研究问答

一句话概括：

**NotebookLM 更偏“基于资料快速理解和问答”，本项目更偏“长期知识沉淀与持续复用”。**

---

### 面试可用表达

“我做这个项目，不是因为市面上没有笔记工具，而是因为我真实遇到了‘收藏即遗忘’和‘看过但不会用’两个问题。  
所以我想做的不是另一个普通笔记产品，而是一个把碎片内容统一收录，再通过内化、图谱和问答沉淀成可复用知识资产的系统。  
和 Obsidian 相比，我不是追求最高自由度，而是希望把默认闭环做得更低门槛；和 NotebookLM 相比，我不是只解决单次资料理解，而是更关注长期知识沉淀和复用。”

## 2026-06-01｜项目开场介绍（第一轮）

### Q1：如果在面试中用 90 秒介绍这个项目，应该怎么讲？

### 结论

这个项目的开场介绍，适合按以下逻辑展开：

1. 先讲用户痛点
- 分散收藏导致“收藏即遗忘”
- 看过内容但没有内化，后续难以复用

2. 再讲产品定义
- 不是单纯笔记工具
- 而是一个把碎片信息沉淀成可复用知识资产的 AI 系统

3. 再讲核心链路
- 收录
- 轻解析
- 内化为结构化笔记
- 形成关系
- 支持可追溯问答与复用

4. 最后讲差异和进展
- 与 Obsidian 的区别：不是比自由度，而是比默认闭环与低门槛
- 与 NotebookLM 的区别：不是只做单次资料理解，而是做长期知识沉淀与复用
- 当前已完成从收录、解析、内化、图谱到 QA 的主链路，并有多轮优化和评测闭环

### 现阶段推荐开场稿

“我做的这个项目，本质上是一个 AI 驱动的个人知识资产系统。它最初来自我自己两个很真实的痛点：第一，我平时在很多平台收藏了大量内容，但它们非常分散，最后经常变成收藏即遗忘；第二，就算后来重新看了，很多内容也只是看过，并没有真正内化成以后能复用的知识。

所以我想做的不是一个单纯存东西的笔记工具，而是一条更完整的知识处理闭环：先把文本、链接、图片、文档这些碎片信息统一收录进来，再做轻解析和知识内化，把它们变成结构化笔记；之后进一步构建知识关系，最后支持基于这些知识做可追溯的问答和复用。

这个项目和传统笔记工具的差异，不在于有没有图谱或者问答这些单点功能，而在于它把‘收录、待处理、内化、关联、问答’做成了一条默认可跑通的产品链路。比如和 Obsidian 相比，我不是追求最高自由度，而是想降低启动和维护成本；和 NotebookLM 相比，我不是只做基于一批资料的智能理解，而是更关注长期的个人知识沉淀和复用。

目前这个项目已经完成了从收录、解析、内化、图谱到 QA 的主链路，也做了多轮优化。尤其在知识内化、图谱关系判定、QA 检索策略、多轮续问和 badcase 回流评测上，已经形成了比较完整的实践闭环。”

### 当前待补强点

1. 还需进一步强化“为什么必须用 AI”
- 否则容易被追问：传统笔记/搜索/标签是否已足够

2. 还需准备一版更偏产品经理口吻的简化版
- 用于非技术面试官场景

## 2026-06-01｜为什么这个产品一定要用 AI（第一轮）

### Q1：为什么不用更强的搜索、标签、便签或表格系统，而一定要引入 AI？

### 结论

如果目标只是“找回内容”，其实不需要把系统做成现在这样。

用户完全可以把链接或材料放进：
- 手机便签
- Excel / 飞书表格
- 普通知识库工具

再通过关键词、标签或筛选做定位。  
这类方案已经能解决一部分“信息存储”和“内容回收”的问题。

但本项目真正想解决的问题不是“找不找得到”，而是：

**找到了之后，能不能直接复用，能不能转化成能力。**

### 为什么搜索不够

1. 搜索只能解决定位，不能解决知识加工
- 它只能把用户带回原始材料
- 用户仍然需要重新阅读、重新理解、重新提炼
- 复用成本依然很高

2. 知识越多，不一定能力越强
- 如果系统只负责存，不负责处理，内容规模上来后会形成认知负担
- 用户会觉得维护成本越来越高，最终放弃使用系统

3. 用户真正需要的是“减负”
- 不是再多一个存储容器
- 而是帮他把原始信息转成之后能直接调用的知识形态

### AI 在这个项目中的必要性

本项目引入 AI，不是为了让搜索“更聪明一点”，而是为了让系统具备知识加工能力，包括：

- 抽取每条内容最重要的信息
- 生成结构化内化笔记
- 区分事实与推断
- 提示可扩展方向
- 逐步帮助用户构建自己的知识架构

因此，AI 的角色不是单点问答增强，而是贯穿“输入 -> 内化 -> 关联 -> 复用”的核心能力层。

### 面试可用表达

“如果我只是想解决‘找回笔记’的问题，其实没必要做成现在这样。我完全可以把链接丢进便签、Excel 或飞书表格，再配合关键词搜索，这已经能解决一部分定位问题。  
但我真正想解决的不是‘找不找得到’，而是‘找到了之后能不能直接复用’。因为原始材料本身往往还是零散的，用户每次都要重新阅读、重新理解、重新提炼，这个成本很高。知识越堆越多，不一定让能力变强，反而可能让系统越来越重，最后用户懒得维护。  
所以我引入 AI，不是为了单纯让搜索更聪明，而是为了让系统具备知识加工能力。它可以先帮我抽取每条内容最重要的信息，做结构化内化，区分事实和推断，再补充可拓展方向，最后把这些内容逐渐组织成我自己的知识架构。这样用户面对的就不再只是原始资料，而是已经被处理过、可以直接调用的知识资产。”

### 可提炼金句

- 传统搜索解决的是“信息回收”，AI 内化解决的是“能力沉淀”。
- 当知识规模变大后，用户的主要矛盾不再是存不下来，而是处理不过来。

## 2026-06-01｜Agent 到底是什么（第一轮）

### Q1：这个项目为什么算 Agent，而不是普通调用大模型？

### 结论

这个项目更适合定义为：

**单 Agent 决策架构，而不是自由自治的多 Agent 系统。**

它的核心不是让模型自由决定所有动作，而是让模型在关键节点承担“决策职责”，再由系统按既定 workflow 执行。

### Agent 在本项目中的职责

在当前项目里，Agent 主要负责：

- 识别用户问题意图
- 判断是否为续问
- 判断是否需要扩图
- 选择关系优先级
- 判断证据是否足够
- 决定回答策略（直接回答 / 部分回答 / 保守回答）

这些都属于“决策层”的工作。

### Workflow 在本项目中的职责

真正执行动作的部分，仍由系统 workflow 完成，例如：

- 按顺序执行检索
- 调用原文切片 / note chunk / graph 扩检索
- 组织证据
- 生成最终回答
- 写日志、埋点、回放、评测

因此，本项目不是“LLM 自由执行”，而是：

**LLM 负责决策，系统负责执行。**

### 为什么这样设计

1. 可控性更强
- 避免模型自由调用造成行为漂移
- 更适合知识问答场景对可信度和边界感的要求

2. 可观测性更强
- 每一层决策都可以单独观察和回放
- badcase 更容易归因到具体环节

3. 更适合迭代
- 可以分别优化意图、检索、扩图、回答
- 不需要每次全链路重写

### 面试可用表达

“我这个项目里的 Agent，不是那种完全自治、自由调用工具的多 Agent 系统，而是一个单 Agent 的决策层。它主要负责在知识处理和问答链路里做关键判断，比如识别用户意图、判断是不是续问、要不要做图谱扩检索、当前证据够不够支撑回答，以及应该采用什么回答策略。  
真正的检索、图谱、回答生成这些执行动作，还是走我设计好的受控 workflow。所以它的本质不是让模型自由发挥，而是让模型负责决策，让系统负责执行。这样做的好处是可控性更强，也更容易做 badcase 归因、评测和持续优化。”

### Q2：既然不是 fully autonomous agent，为什么还叫它 Agent，而不直接说规则工作流？

### 结论

关键区别在于：

- **工作流**解决的是“怎么执行”的问题
- **Agent**解决的是“先判断该怎么走”的问题

本项目不是纯规则工作流，因为真实用户提问并不是标准化输入。

### 为什么不能只说是工作流

在真实场景里，用户问题具有以下特点：

- 表达方式多样
- 问题目标不固定
- 多轮上下文有连续性
- 是否扩图、是否保守回答、是否复用上轮语境，都不是单一固定流程能覆盖的

因此，系统不能只依赖固定 if-else 规则直接往下执行，而是需要一个能基于语义和上下文做动态判断的决策层。

### 为什么又不是 fully autonomous agent

虽然前面需要模型做决策，但后面的执行仍然需要受控：

- 知识问答对可信度要求高
- 需要可回放、可观测、可评测
- badcase 需要精确归因
- 不能让模型自由决定整个动作序列

因此，本项目不是“完全放权给模型”，而是把模型限制在决策层，把执行留给 workflow。

### 更准确的架构表达

**Agent 决策 + Workflow 执行**

也就是：

- 在不确定处用模型
- 在确定处用流程

### 面试可用表达

“因为工作流解决的是执行确定性的问题，但用户输入并不是确定性的。真实用户提问会很复杂，表达方式、问题目标、上下文连续性都不一样，所以系统不能只靠一套固定规则直接往下跑。它需要先有一个决策层去判断：用户到底在问什么、是不是在追问上一个话题、当前应该优先走哪种检索路径、要不要扩图、证据够不够回答。  
这些判断本质上不是简单 if-else 能完全覆盖的，而是需要模型基于语义和上下文做动态决策。  
所以我不把它定义成纯规则工作流，因为它确实有决策智能；但我也不把它定义成完全自治 Agent，因为后续执行仍然是受控 workflow。更准确地说，它是一个‘Agent 决策 + Workflow 执行’的架构。”

### 可提炼金句

- 工作流负责“怎么执行”，Agent 负责“先判断该怎么走”。
- 在不确定处用模型，在确定处用流程。
- 这个项目的 Agent 性，不体现在“会不会自己乱跑”，而体现在“会不会根据语义和上下文做决策”。

### Q3：这个项目里的 Agent 具体做了哪些决策？

### 结论

这个项目里的 Agent 主要不是直接生成最终答案，而是承担 QA 链路中的连续决策职责。

它更像一个**调度与判断中枢**，主要负责三类决策：

#### 1. 前置路由决策

在问答开始前，先判断用户问题属于什么类型，例如：

- 事实查询
- 总结归纳
- 对比决策
- 执行建议
- 复盘反思

然后根据意图决定：
- 优先走哪类检索路径
- 使用怎样的检索参数
- 是否倾向某类关系扩展

#### 2. 检索中的动态决策

在检索过程中，不是固定一路跑到底，而是持续根据当前证据状态做判断，例如：

- 当前证据是否足够
- 是否为续问
- 是否需要复用上轮上下文
- 是否需要扩图
- 是否需要继续召回
- 何时停止检索

这说明 Agent 不是单次判断，而是**贯穿检索过程的连续决策层**。

#### 3. 回答阶段决策

在拿到证据后，Agent 还需要参与回答策略判断，例如：

- 先对检索结果做逻辑化整理
- 判断当前适合直接回答、部分回答还是保守回答
- 选择更合适的回答方式输出

因此，回答不是简单把召回内容拼接出来，而是经过策略判断后的结果。

### 更准确的总结

**这个 Agent 不直接替用户回答所有问题，它更像是 QA 链路里的调度与判断中枢。**

### 面试可用表达

“在我这个项目里，Agent 主要不是直接写答案，而是负责整条 QA 链路里的关键决策。  
第一类是前置路由决策，比如先识别用户意图，判断这是事实查询、总结、对比、行动建议还是复盘，再决定优先走哪种检索路径和检索参数。  
第二类是检索中的动态决策，比如根据当前召回结果判断证据够不够、是不是续问、要不要复用上轮上下文、要不要做图谱扩检索，以及什么时候继续召回、什么时候停止。  
第三类是回答阶段决策，也就是在证据拿到之后，不是直接拼接输出，而是先做逻辑整理，再判断这题适合直接回答、部分回答还是更保守地回答。  
所以它的作用更像是一个调度和判断中枢，而不是一个完全自由发挥的大模型助手。”

### 可提炼金句

- 这个 Agent 不直接替用户回答所有问题，它更像 QA 链路里的调度与判断中枢。
- 它不是单次分类器，而是贯穿问答过程的连续决策层。
- 真正的差异不在“会不会回答”，而在“会不会根据证据状态动态调整策略”。

## 2026-06-02｜意图识别怎么设计（第一轮）

### Q1：这个项目里的意图识别是怎么设计的？

### 结论

本项目的意图识别，不适合理解为一个简单分类器，而更适合理解为：

**QA 决策层的第一道路由器。**

它的目标不是只给问题打一个标签，而是为后续检索、扩图和回答策略提供决策依据。

### 当前采用的三层结构

#### 1. 主意图

主意图描述的是用户这一轮问题的主要任务目标，例如：

- 事实查询
- 总结归纳
- 对比决策
- 执行建议
- 复盘反思

这一层主要决定：

- 检索参数怎么设
- 检索更偏精准还是偏覆盖
- 回答应更偏事实、归纳、对比还是建议

#### 2. 扩展意图

扩展意图不是判断“这是什么问题”，而是判断：

**如果当前证据不够，系统应该优先往哪个方向扩。**

当前项目中的扩展意图包括：

- `evidence_strengthening`
- `example_request`
- `related_topic_expansion`

它们分别对应：

- 补支撑证据
- 找案例或例子
- 扩相关主题

这一层主要决定：

- 图谱扩检索时关系类型优先级怎么排
- 在 `supports` / `example_of` / `related` 之间如何偏置

#### 3. 连续性门控（续问识别）

这一层不是问题类型分类，而是识别：

**当前问题是不是在延续上一轮上下文。**

因为真实用户常常不会完整重述问题，而会直接说：

- 继续
- 上一个话题
- 再展开一下
- 那这个怎么做

如果识别不到这一层，系统就很容易在多轮问答中断链或跑偏。

### 为什么不只做单标签分类

因为单一标签不足以支撑后续动作。

例如一句话：

“继续上一个 PRD 话题，给我一个最小模板”

它可能同时包含：

- 主意图：执行建议
- 扩展意图：案例/模板请求
- 连续性：明确续问

如果只输出一个总标签，后续检索与扩图策略会过于粗糙。

### 更准确的总结

**本项目的意图识别是“主意图 + 扩展意图 + 连续性门控”的三层结构，而不是单层分类。**

### 面试可用表达

“我这个项目里的意图识别不是单层分类，而是 QA 决策层的第一道路由器。  
第一层是主意图，判断用户这轮问题更像事实查询、总结、对比、建议还是复盘，这会直接影响检索参数和回答方式。  
第二层是扩展意图，它不是判断问题属于哪一类，而是判断如果当前证据不够，系统更应该往哪个方向扩，比如是补支撑证据、找例子，还是扩相关主题，这会影响图谱扩检索时关系优先级的选择。  
第三层是连续性判断，也就是识别这是不是接着上一轮在问。因为真实用户经常不会把问题重说一遍，而是直接说‘继续’、‘展开一下’，如果识别不到这一层，检索就很容易断链。  
所以我们实际上做的是‘主意图 + 扩展意图 + 连续性门控’三层结构，而不是简单打一类标签。”

### 可提炼金句

- 意图识别不是分类器，而是 QA 决策层的第一道路由器。
- 不是为了分类更漂亮，而是为了让每个分类结果都对后续动作有用。
- 单标签只能描述问题，三层结构才能驱动系统行为。

### Q2：为什么把主意图设计成这 5 类：事实查询、总结归纳、对比决策、执行建议、复盘反思？

### 结论

这 5 类主意图不是从技术实现方便出发随意列的，而是从用户最常见的知识使用场景倒推出来的。

核心思路是：

**用户收藏内容，不只是为了“存起来”，而是为了后续以不同方式使用它。**

在当前项目里，主意图的分类逻辑主要对应五类高频使用需求。

#### 1. 事实查询

这是最基础的使用场景。

用户收藏一条内容后，后续最先会问的是：

- 这条内容到底讲了什么
- 某个概念、方法、结论是什么
- 某个信息有没有被提到

这一类问题的核心是“找准并说清楚”。

#### 2. 总结归纳

当知识内容变多后，用户不会满足于单点信息定位，而会进一步希望系统帮他：

- 提炼重点
- 做结构化总结
- 把零散信息压缩成可快速吸收的形式

这一类问题的核心是“从信息中提炼结构”。

#### 3. 对比决策

知识系统不只是回忆工具，还应该支持判断和选择。

用户在真实场景里会问：

- A 和 B 有什么区别
- 哪种方法更适合当前场景
- 两种方案的优缺点和取舍是什么

这一类问题的核心是“基于已有知识做横向比较和辅助决策”。

#### 4. 执行建议

很多用户收藏“学习干货”和“工作方法”，最终并不是只想理解，而是想落地。

典型问题包括：

- 这个方法怎么用到工作中
- 我下一步应该怎么做
- 给我一个可执行模板或最小实践方案

这一类问题的核心是“把知识转成行动”。

#### 5. 复盘反思

知识系统如果只支持查和用，还不够体现“体系化”价值。

更进一步，系统还应该帮助用户：

- 回看过去的做法有什么问题
- 复盘某次实践的不足
- 找出优化点和经验教训

这一类问题的核心是“利用已有知识做反思与迭代”。

### 为什么这 5 类适合当前项目

因为这 5 类基本覆盖了用户从“知道”到“理解”到“比较”到“行动”再到“复盘”的主要链路。

也就是说，它不是一个静态知识库视角的分类，而是一个**面向知识使用过程**的分类。

这也和项目定位一致：

- 项目不是只帮用户找资料
- 也不是只帮用户做摘要
- 而是希望帮助用户基于自己的知识资产完成理解、决策、执行和反思

因此，这 5 类主意图本质上是在服务“知识复用”这件事，而不是服务分类本身。

### 面试可用表达

“这 5 类主意图不是我从模型任务角度硬拆出来的，而是从用户最常见的知识使用场景倒推的。  
最基础的是事实查询，也就是用户想先知道这条内容到底讲了什么；然后是总结归纳，帮助用户快速提炼重点；再往上是对比决策，因为知识系统不只是回忆工具，还要支持判断和选择；再往后是执行建议，很多用户收藏干货最终是想落地到学习、工作和生活里；最后是复盘反思，因为一个好的知识系统不应该只支持查和用，还应该帮助用户基于已有知识做反思和迭代。  
所以这 5 类本质上覆盖的是用户从‘知道’到‘理解’到‘比较’到‘行动’再到‘复盘’的知识使用链路，它们服务的是知识复用，而不是分类本身。”

### 可提炼金句

- 这 5 类不是从模型任务出发拆的，而是从用户知识使用场景倒推出来的。
- 它覆盖的是用户从“知道”到“行动”再到“复盘”的知识使用链路。
- 主意图分类服务的是知识复用，而不是分类本身。

### Q3：面试中如何按“意图体系 -> 数据 -> 模型 -> 槽位 -> 不确定性”讲清楚意图识别？

### 面试表达版

“我做意图识别时，没有把它简单当成一个文本分类任务，而是按线上意图系统的思路拆了五步。

第一步是定义意图体系。  
我的项目是个人知识系统，所以核心意图不是订单里的查询、退款、取消，而是围绕知识复用来定义的。我把主意图分成 5 类：事实查询、总结归纳、对比决策、执行建议、复盘反思。  
这 5 类对应用户从‘知道一条内容讲什么’，到‘提炼重点’，再到‘比较判断’，最后到‘怎么落地’和‘怎么复盘优化’的使用链路。除此之外，我还单独定义了扩展意图，比如要证据、要案例、要相关主题，因为这些会影响后续图谱扩检索方向。

第二步是准备和沉淀样本。  
我会给每类意图准备不同表达，比如执行建议不只包括‘怎么做’，还包括‘下一步怎么落地’、‘给我一个模板’、‘这个方法怎么用到面试里’；对比意图不只包括‘对比’，也包括‘区别是什么’、‘哪个更适合’、‘怎么取舍’。  
我会特别补口语、省略、多意图和混淆表达。比如‘为什么这个方法有效’既可能是事实查询，也可能是要补充支撑证据；‘继续上一个 PRD 话题，给我一个最小模板’同时包含续问、PRD 槽位和模板请求。这个 case 在我的项目里就作为过 badcase。

第三步是模型和路由选择。  
我没有一开始就完全依赖大模型，因为关键决策如果全交给 LLM，会有漂移和不可复现的问题。冷启动阶段我采用规则关键词做可控基线，再让 LLM 决策层参与复杂判断。  
比如主意图先用规则识别，扩展意图也有明确关键词触发：要依据、证明、原因时偏 `supports`；要案例、模板时偏 `example_of`；要相关、延伸时偏 `related`。后续如果样本量上来，我会升级成规则 + 分类模型 + LLM 组合路由，而不是单一路径。

第四步是槽位抽取。  
意图只说明用户想做什么，槽位才说明系统怎么执行。我的项目里槽位不是城市、日期这种通用槽位，而是知识系统里的执行参数，比如主题、来源、标签、时间范围、上一轮上下文、目标对象。  
例如用户问‘继续上一个 PRD 话题，给我一个最小模板’，里面的意图是执行建议，扩展意图是模板请求，槽位包括 `topic=PRD`、`template_type=最小模板`、`followup=true`、`context=上一轮命中的 PRD 笔记`。这样系统才能知道不是重新全库搜索‘模板’，而是优先锚定上轮 PRD 笔记。

第五步是处理不确定性。  
我把结果分成高、中、低置信度来处理。高置信度就直接进入检索；中置信度可以结合上下文或追问澄清；低置信度就保守回答或提示证据不足。  
在我的 QA 链路里，也对应了 `answerable / partial / insufficient` 三档回答策略。比如证据强就直接回答，证据中等就部分回答并说明边界，证据不足就不强行编。这个设计的核心是让系统可控，避免为了回答而回答。

我实际验证时用了 7 题 QA 回归，覆盖单轮事实查询、总结、多轮续问、跨话题切换和无关联判断。一个典型 badcase 是：用户先问 PRD 模板包含哪些模块，下一轮问‘继续上一个 PRD 话题，给我一个最小模板’，早期系统没有把它识别成强续问，结果召回到了 Agent 设计笔记。后来我加强了续问识别和上轮 note 复用，第三轮复测里这个 case 通过，整体 7 题也通过。

所以我对意图识别的理解是：它不只是文本分类，真正影响线上效果的是意图边界、槽位抽取、置信度管理、澄清策略和 badcase 持续迭代。我的项目里，意图识别的价值不是‘打对标签’，而是驱动后面的 RAG 检索路径、图谱扩展方向和回答策略。”

### 30 秒压缩版

“我做意图识别时按线上系统思路拆了五步：先定义意图体系，再准备不同表达样本，然后选择规则 + LLM 决策的冷启动路由，同时抽取知识场景里的槽位，最后用置信度和证据强弱决定直接回答、澄清还是保守回答。  
我的项目里主意图是事实查询、总结、对比、建议、复盘，扩展意图是要证据、要案例、要相关主题。比如‘继续上一个 PRD 话题，给我最小模板’，它不是单一意图，而是续问 + PRD 主题 + 模板请求。之前这个 case 召回跑偏，后来通过续问识别和上轮 note 复用修复，7 题 QA 回归里已经通过。  
所以我理解意图识别不是分类本身，而是为了驱动 RAG 路径、槽位执行和回答策略。”

### 可提炼金句

- 意图识别不是分类本身，而是为了驱动 RAG 路径、槽位执行和回答策略。
- 意图说明用户要做什么，槽位说明系统怎么执行。
- 关键决策不能完全依赖 LLM，需要规则、模型、上下文和置信度共同约束。

## 2026-06-02｜为什么个人知识系统也需要 RAG

### Q1：RAG 一般用于企业内部知识库，个人知识系统数据不多，为什么也要用？

### 结论

RAG 的必要性不只由数据规模决定，而由三个因素决定：

1. 知识是不是私有的
2. 知识是不是动态变化的
3. 回答是不是需要可追溯

企业知识库是 RAG 的典型场景，但个人知识库也有轻量 RAG 的合理性。

### 为什么个人知识系统也需要 RAG

#### 1. 私有性

用户自己的收藏、笔记、理解和复盘记录，并不在通用大模型参数里。

如果不检索这些材料，模型只能基于通用知识回答，无法真正回答：

**“我的知识库里有什么？”**

#### 2. 动态性

个人知识是持续变化的。

用户今天收藏一篇文章，明天内化一条笔记，后天又形成新的知识关系。  
这些内容不会实时进入模型参数，只能通过检索接入回答链路。

#### 3. 可追溯性

本项目强调的是可信问答，而不是泛泛聊天。

用户不仅需要答案，还需要知道：

- 答案依据来自哪条笔记
- 是否有原文证据
- 当前证据够不够
- 哪些地方需要保守表达

这就需要 RAG 提供引用、证据边界和可回放能力。

### 为什么不是过度设计

如果只是十几条固定笔记的 demo，确实可以先用关键词搜索或全文搜索。  
但本项目的目标不是短期找几条内容，而是验证一条长期知识复用链路。

所以当前做的是：

**个人轻量 RAG，而不是企业级重 RAG。**

它的核心能力包括：

- chunk
- embedding
- 召回
- 引用
- 证据边界
- badcase 回放

这样既能支撑当前规模，也能为后续数据增长、多轮问答和评测闭环留下空间。

### 面试可用表达

“我理解 RAG 最典型的场景确实是企业内部知识库，因为企业文档多、知识私有、模型本身不知道。但我觉得 RAG 的必要性不只由数据规模决定，而是由三个因素决定：知识是不是私有的、是不是动态变化的、回答是不是需要可追溯。  
我的项目虽然是个人知识系统，数据量没有企业那么大，但用户收藏的内容、自己的理解、内化笔记，本质上也是私有知识，而且会持续更新。大模型如果不检索这些材料，就只能基于通用知识回答，无法真正回答‘我的知识库里有什么’。  
所以我这里做的不是企业级重 RAG，而是个人轻量 RAG。它的作用是让回答基于用户自己的笔记和原文证据，减少幻觉，并且能返回引用和证据边界。数据少的时候可以从简单检索开始，但只要目标是长期知识复用和可追溯问答，RAG 这层就是合理的。”

### 可提炼金句

- RAG 的必要性不只由数据规模决定，而由私有性、动态性和可追溯性决定。
- 企业知识库是 RAG 的典型场景，个人知识库是 RAG 的轻量场景。
- 我不是为了做 RAG 而做 RAG，而是为了让模型基于我的知识资产回答，而不是基于通用常识发挥。

## 2026-06-02｜为什么同时检索内化笔记和原文切片

### Q1：为什么不只检索一种内容，而要同时有内化笔记检索和原文切片检索？

### 结论

内化笔记和原文切片解决的是两个不同问题：

- **内化笔记**更像知识索引和理解层，负责快速定位主题、结构和可复用观点。
- **原文切片**更像证据层，负责提供更高密度、更贴近原始材料的事实依据。

所以两者不是重复，而是互补。

### 内化笔记的作用

内化笔记的目标不是复制原文，而是把原始材料转成可复用知识。

它通常包含：

- 重要内容概括
- 关键概念
- 原文支持要点
- 推断与延展
- 可关联方向

因此，内化笔记适合承担第一层检索：

- 更容易快速判断主题范围
- 更容易命中用户的抽象问题
- 能利用摘要、概念、标签和关联方向加速定位
- 适合回答总结、建议、复盘这类高层问题

### 原文切片的作用

原文切片保留的是更接近原始材料的内容。

它的优势是：

- 信息密度更高
- 事实细节更完整
- 更适合做证据引用
- 能减少内化过程中的概括损失

因此，原文切片适合承担第二层精确定位：

- 在内化笔记确定主题范围后，进一步找具体依据
- 支撑事实查询和引用
- 帮助回答层减少幻觉

### 两者如何配合

更理想的链路是：

1. 先用内化笔记快速确定知识范围
2. 再在相关原文切片里定位具体证据
3. 最终回答时同时利用“结构化理解”和“原文依据”

这样可以同时兼顾：

- 速度
- 召回范围
- 事实准确率
- 证据可追溯性

一句话概括：

**内化笔记负责让系统更懂，原文切片负责让回答更准。**

### 面试可用表达

“我没有只做一种检索，是因为内化笔记和原文切片承担的职责不一样。  
内化笔记的目标是把原始材料转成可复用知识，它里面有重点概括、关键概念、事实和推断分层，还有可关联方向，所以它更适合作为第一层知识索引，帮助系统快速确定这次问题大概落在哪些知识范围里。  
但内化笔记毕竟是经过模型加工后的结果，可能会有概括损失，所以还需要原文切片作为证据层。原文切片信息密度更高，更贴近原始材料，适合做事实定位和引用。  
所以我的设计是先通过内化笔记快速缩小范围，再用原文切片精准定位证据，最后回答时同时利用结构化理解和原文依据。这样能在速度、准确率和可追溯性之间做平衡。”

### 可提炼金句

- 内化笔记是理解层，原文切片是证据层。
- 内化笔记负责让系统更懂，原文切片负责让回答更准。
- 先用内化笔记确定范围，再用原文切片定位证据，可以同时兼顾速度和准确率。

## 2026-06-02｜图谱为什么要接入 RAG

### Q1：图谱在 QA 里到底起什么作用？

### 结论

图谱在 QA 里承担的是**二阶段扩检索能力**，不是为了可视化好看，也不是替代向量检索。

RAG 的第一步仍然是从内化笔记和原文切片里找直接证据。  
只有当直接证据不够，或者用户的问题天然需要关联扩展时，才用图谱沿着知识关系补充相邻内容。

### 为什么图谱不是第一入口

用户问一个问题时，最可靠的证据通常还是直接命中的内化笔记和原文切片。

如果一开始就用图谱扩展，很容易把“相关但不必要”的内容带进来，导致：

- 引用噪音变多
- 回答重点发散
- token 成本上升
- badcase 更难归因

所以本项目把图谱放在第二阶段，而不是第一入口。

### 图谱扩检索适合解决什么问题

图谱接入 RAG 的价值，是补充向量检索的不足。

向量检索更适合找语义相似内容，而图谱更适合沿着关系找：

- 支撑证据
- 案例模板
- 相关主题
- 相邻知识

因此，图谱更适合作为“证据不足时的补充路径”。

### 什么时候扩图

主要有三类场景：

1. 直接证据不足
- 召回片段太少
- 证据强度不够
- 当前回答只能 partial 或 insufficient

2. 用户明确要扩展
- “有没有案例”
- “为什么这个方法成立”
- “还有什么相关内容”
- “能不能延伸一下”

3. 多轮问答需要上下文延展
- 用户沿着上一个话题继续追问
- 可以从上一轮命中的 note 出发，扩展到相邻知识

### 扩图方向怎么决定

图谱扩检索不是盲目扩，而是由扩展意图决定关系优先级：

- 用户要依据、原因、证明：优先 `supports`
- 用户要例子、模板、案例：优先 `example_of`
- 用户要相关、延伸、类似主题：优先 `related`

这说明图谱扩检索和前面的意图识别是连在一起的。

### 什么时候停止扩图

图谱扩检索必须“够用即停”，不能为了扩而扩。

常见停止条件包括：

- 扩展预算达到上限
- 跳数达到上限
- 没有可用关系边
- 新增证据收益不够
- 已经有足够证据回答

### 面试可用表达

“图谱接入 RAG 的价值，不是替代向量检索，而是补充向量检索的不足。向量检索更适合找语义相似内容，但有时用户的问题需要沿着知识关系扩展，比如要一个案例、要支撑依据，或者想看相关主题。  
所以我把图谱放在二阶段扩检索里：先用内化笔记和原文切片找直接证据；如果证据不足，或者用户意图显示需要例子、依据、相关主题，再触发图谱扩展。扩展时也不是盲目扩，而是根据扩展意图选择关系类型，比如要证据优先走 `supports`，要案例优先走 `example_of`，要相关主题优先走 `related`。  
同时我也设计了停止规则，比如预算、跳数、最小收益、无可用关系边，以及证据已经够用就停止。这样图谱在 QA 里是一个受控的扩展层，而不是噪音放大器。”

### 可提炼金句

- 图谱不是 QA 的第一入口，而是二阶段扩检索能力。
- 向量检索找相似，图谱扩检索找关系。
- 图谱扩检索要有方向，也要够用即停。
- 图谱的价值不是让回答更“多”，而是让证据不足时有更合理的补充路径。

## 2026-06-02｜RAG 检索链路完整流程

### Q1：你的 RAG 检索链路完整流程是什么？

### 结论

本项目的 RAG 不是单一步向量召回，而是一个逐级补证的分层链路：

`问题理解 -> 意图识别 -> 内化笔记检索 -> 原文切片检索 -> 续问复用 -> 图谱扩检索 -> 证据加工 -> 回答策略`

这条链路的核心原则是：

**先找最直接、最可信、最少噪音的证据；不够时，再逐级扩展。**

### 1. 问题理解与意图识别

先判断用户问题属于：

- 事实查询
- 总结归纳
- 对比决策
- 执行建议
- 复盘反思

同时判断：

- 是否为续问
- 是否需要案例
- 是否需要支撑证据
- 是否需要相关主题扩展

这一层决定后续检索参数和扩展方向。

### 2. 内化笔记检索

先检索内化笔记，因为它是结构化后的知识理解层。

它适合：

- 快速定位主题范围
- 命中抽象问题
- 支撑总结、建议、复盘类问题

### 3. 原文切片检索

再检索原文切片，因为它更接近原始材料。

它适合：

- 提供事实证据
- 做引用
- 减少内化概括带来的信息损失

### 4. 续问复用

多轮问答中，如果用户是在延续上轮话题，则优先复用上一轮命中的 note。

这样可以降低：

- 重新全库检索导致跑偏
- 同主题续问断链
- 上下文丢失

### 5. 图谱扩检索

当直接证据不足，或者用户明确要案例、依据、相关主题时，再触发图谱扩检索。

图谱扩检索用于补充：

- `supports`
- `example_of`
- `related`

但必须受预算、跳数和收益阈值约束。

### 6. 证据加工与回答策略

拿到证据后，不是直接拼接给模型，而是先做：

- 去重
- 逻辑梳理
- 冲突检查
- 证据强弱判断

然后选择：

- `answerable`
- `partial`
- `insufficient`

### 面试可用表达

“我的 RAG 不是单一步向量召回，而是一个逐级补证的分层链路。  
第一步先做问题理解和意图识别，判断用户是在问事实、总结、对比、建议还是复盘，同时判断是不是续问、是否需要案例或支撑证据。  
第二步先检索内化笔记，因为内化笔记是结构化后的知识层，适合快速判断主题范围。  
第三步再检索原文切片，因为原文切片更接近原始材料，适合提供事实证据和引用。  
如果是多轮场景，还会做续问复用，优先使用上一轮命中的 note，避免因为重新全库检索导致跑偏。  
最后，如果直接证据还不够，或者用户明确要案例、依据、相关主题，再触发图谱扩检索，沿着 supports、example_of、related 这些关系补证。  
拿到证据后，不是直接拼接给模型，而是先做证据加工和去噪，再根据证据强弱决定是直接回答、部分回答还是保守回答。”

### 可提炼金句

- 这条链路的核心不是“尽可能多召回”，而是“按证据可靠性逐级补足”。
- 先直接证据，再扩展证据；先控制噪音，再补召回。
- 检索到内容不等于可以直接回答，还要做证据加工和边界判断。

## 2026-06-02｜RAG 怎么评测

### Q1：怎么判断 RAG 检索是好是坏？

### 结论

RAG 评测不能只看最终回答是否正确，而要拆成三层：

1. 检索层
2. 证据层
3. 回答层

因为一个 QA badcase 可能来自：

- 意图识别错
- 检索没命中
- 扩图带噪音
- 证据不足
- 回答模型组织错

所以评测目标是：

**分层评测，分层归因。**

### 1. 检索层评估

检索层看系统有没有找对内容。

可用指标：

- `hit_rate@k`
- `chunk_hit_rate`
- `followup_hit_rate`
- `retrieval_stage`

重点问题：

- 正确 note 有没有进入 topK
- 正确 chunk 有没有被召回
- 续问是否命中上轮上下文
- 最终证据来自内化、原文还是图谱

### 2. 证据层评估

证据层看召回内容是否能支撑回答。

可用指标：

- `citation_relevance`
- `evidence_strength`
- `noise_rate`
- `graph_expand_gain`
- `wrong_expand_rate`

重点问题：

- 引用是否真的相关
- 是否有低相关噪音
- 图谱扩检索是否真的带来有效证据
- 是否应该提前截断低证据片段

### 3. 回答层评估

回答层看模型是否正确使用证据。

可用指标：

- `answer_accuracy`
- `faithfulness`
- `boundary_quality`
- `user_task_success`

重点问题：

- 答案是否正确
- 是否忠于证据
- 是否有幻觉
- 证据不足时是否表达边界

### 典型 badcase 1：Q5 续问断链

现象：

- 用户先问 PRD 模板包含哪些模块
- 下一轮问“继续上一个 PRD 话题，给我一个最小模板”
- 早期系统召回到了 Agent 设计笔记

归因：

- 不是回答模型问题
- 是续问识别和检索锚定问题
- 主检索重新全库召回时被“模板/方法论”语义带偏

修复：

- 加强续问识别
- 复用上一轮命中的 note
- 对续问场景做强锚定

结果：

- 第三轮 7 题 QA 复测中 Q5 通过

### 典型 badcase 2：Q7 无关联判断引用噪音

现象：

- 用户问“意图识别和 PRD 有什么关联”
- 回答能判断“暂无直接关联”
- 但早期召回和引用存在噪音

归因：

- 回答层判断基本正确
- 检索层和证据层仍有低相关片段进入上下文

修复方向：

- 低证据分提前截断
- citation 去噪
- 无关联问题单独策略

### 面试可用表达

“我评测 RAG 时会拆成检索、证据、回答三层，而不是只看最终答案。  
检索层看正确笔记和 chunk 有没有召回，比如 hit_rate@k、chunk 命中率、续问是否命中上轮上下文。  
证据层看召回内容是否真的能支撑回答，比如引用相关性、噪音比例、证据强度、图谱扩检索是否带来有效增益。  
回答层再看答案是否准确、是否忠于证据、证据不足时有没有合理边界。  
比如我项目里有个 Q5 badcase，用户先问 PRD 模板，下一轮说继续上一个 PRD 话题给最小模板，早期系统召回到了 Agent 设计笔记。这个问题不是回答模型错，而是续问锚定和检索召回错，所以我通过续问识别和上轮 note 复用修复，第三轮 7 题回归里这个 case 通过。  
还有一个 Q7 无关联判断，回答能说暂无直接关联，但早期引用噪音较多，这说明最终答案对了不代表检索层就好，所以后续要单独看 citation relevance 和低相关噪音率。”

### 可提炼金句

- RAG 评测要分层：检索层、证据层、回答层。
- 最终答案对了，不代表检索就是好的。
- 一个 badcase 要能归因到意图、检索、扩图、证据或回答生成。

## 2026-06-08｜项目 API 选型的面试表达（第一轮）

### Q1：如果面试官问“你这个项目都调用了哪些 API，为什么这么选”，应该怎么回答？

### 结论

这个问题不适合答成“我接了哪些接口”的工程清单，更适合按 AI 产品链路去讲。  
更好的回答方式是：

1. 先讲完整链路
- 信息收录
- 内容解析
- 知识内化
- 向量检索
- 问答输出

2. 再讲每一段用了什么 API
- `Supabase`
- `LLM 文本模型`
- `Embedding API`
- `OCR / 多模态 API`
- `PDF 解析服务`
- `网页抓取与在线知识源`

3. 最后讲为什么这样拆，而不是一个模型或一个平台包打天下

这个项目最适合被概括成：

**不是“接了一个大模型接口”的 AI 产品，而是把不同任务拆给最合适能力层的一条完整知识处理链路。**

### 这道题的推荐回答逻辑

#### 1. 先讲产品链路，不要先讲技术名词

面试里先把项目说成一条链路：

**信息收录 -> 内容解析 -> 内化生成 -> 向量检索 -> 问答输出**

这样做的好处是，后面每个 API 都有明确位置，而不是零散罗列。

#### 2. 再讲 API 分层

这个项目里的 API 主要分成下面几类：

- `Supabase`
  - 负责认证、数据库、文件存储、向量检索相关 RPC。
- `LLM 文本模型`
  - 负责摘要、标签、笔记生成、关系判断、问答生成。
- `Embedding API`
  - 负责把文本转成向量，用于检索召回。
- `OCR / 多模态 API`
  - 负责图片识别、图片摘要。
- `PDF 解析服务`
  - 负责 PDF 文本提取，复杂场景走 Python sidecar。
- `网页抓取能力`
  - 负责解析普通网页、微信、小红书内容。
- `在线知识源 API`
  - 在 QA 场景下补充 Wikipedia、arXiv 等外部知识。

#### 3. 面试重点不在“怎么接”，而在“为什么分开接”

这道题真正想考的，往往不是工程接法，而是你是否理解 AI 产品的能力拆分。

核心逻辑是：

- `Supabase` 用来承接 `Auth + Postgres + pgvector + RLS + Storage`
  - 因为 AI 产品早期更需要低集成成本和快速迭代，而不是把基础设施拆得很散。
- `文本生成` 默认走 LLM chat completions
  - 因为摘要、标签、笔记生成、问答，本质都是生成与理解任务。
- `Embedding` 单独拆出去
  - 因为“生成模型表现好”不等于“向量检索效果最好”，召回质量要单独优化。
- `OCR / 多模态` 单独拆出去
  - 因为图片识别和纯文本生成不是同一类能力。
- `PDF 解析` 单独走 Python sidecar
  - 因为复杂文档解析在 Python 生态更成熟，稳定性更高。
- `网页抓取` 做多级兜底
  - 因为真实产品里，内容获取成功率比“只保留一个优雅方案”更重要。

### 一版适合面试的答案

“我这个项目里的 API 设计，不是一个大模型接口解决所有问题，而是按产品链路拆开的。整体链路是信息收录、内容解析、知识内化、检索召回、最后问答输出。

第一层是基础设施，我用的是 Supabase。它承担了用户认证、业务数据存储、文件存储，以及向量检索相关能力。之所以选它，而不是自己拼 Auth、数据库、对象存储和向量库，是因为 AI 产品早期更看重低集成成本和快速迭代，Supabase 这种一体化方案能明显降低复杂度，而且 RLS 也能保证多用户数据隔离。

第二层是模型能力，但我没有把所有任务都交给同一个模型。文本生成任务，比如摘要、标签、内化笔记、问答生成，主要走 LLM 的 chat completions 接口；向量检索单独走 embedding 接口；图片 OCR 和多模态理解单独走视觉模型接口。这样拆的原因是，不同任务的最优模型不一样，文本模型、向量模型和多模态模型应该分别选型，而不是强行统一。

第三层是解析能力。像 PDF 这种复杂文档，我没有完全依赖前端或 Node 侧处理，而是单独接了一个 Python 的 parser service。因为文档解析、OCR、版面恢复这些能力在 Python 生态更成熟，稳定性更高。这个选择本质上是为了保证内容进入知识库之前，原始信息提取的质量足够高。

第四层是内容获取。普通网页可以直接抓 HTML，但像微信、小红书这种平台反爬比较强，所以项目里做了多层策略，比如正文抓取、渲染页兜底、图片 OCR、外部 crawler 可切换。这里体现的不是单个 API 的能力，而是产品设计里对成功率和可维护性的平衡。

如果从产品视角总结，这套 API 方案背后的原则不是‘选最强模型’，而是‘按任务选最合适的能力’，同时保证链路可替换、可降级、可追踪。这对 AI 产品很重要，因为真正上线后，成本、稳定性、召回质量、内容解析成功率，往往比单点模型能力更决定体验。”

### 面试可用总结句

- 这个项目的 API 选型不是单点最强，而是任务分治。
- 真正的产品思路不是“接一个大模型”，而是搭一条可替换、可降级、可持续优化的 AI 能力链。
- 对 AI 产品来说，成本、稳定性、召回质量和解析成功率，往往比单一模型能力更决定用户体验。
