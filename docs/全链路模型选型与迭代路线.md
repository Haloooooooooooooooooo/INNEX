# 全链路模型选型与迭代路线

日期：2026-05-28  
适用范围：INNEX 从录入到 QA 的全链路模型治理

## 1. 目标与原则

目标：不是一次性选“最强模型”，而是为不同任务选“最合适模型”，并形成可迭代的路由策略。  
原则：
- 任务分治：按 useCase（parse/internalize/embedding/qa）分别选型。
- 阶段闭环：每个 Phase 只做当期必要评测，避免大而全阻塞开发。
- 数据驱动：统一样本、统一指标、统一成本口径。
- 可回退：每个关键任务必须有主模型和降级策略。

## 2. 范围拆解（必须覆盖）

当前纳入选型范围：
1. 录入与解析（capture/parse）
2. 内化（internalize）
3. 向量化（embedding）
4. QA 检索与回答（qa）
5. 图谱关系判定（graph relation，Phase4起重点）

说明：录入必须纳入。它是上游入口质量，直接决定内化与QA上限。

## 3. 选型评估框架（统一模板）

### 3.1 场景需求拆解
- 任务类型：生成式、理解式、多模态。
- 时延目标：
  - 交互式（QA）建议 P95 < 2.5s（首包）
  - 后台任务（internalize）建议单任务 < 30s
- 输入输出：最大上下文、结构化输出稳定性、中文能力、多语言能力。

### 3.2 技术参数评估
- 模型能力：通用推理、长文本理解、结构化遵循（JSON/Markdown）。
- 公开基准：MMLU/CMMLU/long-context/工具调用能力（仅参考，不直接替代业务评测）。
- 任务拟合：是否适配“事实优先、证据可追溯”的产品要求。

### 3.3 成本效益分析
- API 成本：输入/输出 token 单价，重试放大系数。
- 性能成本：平均耗时、P95、失败重试率。
- 隐性成本：监控、告警、供应商变更、版本漂移。

### 3.4 工程化适配度
- 接入复杂度：SDK 兼容、鉴权方式、限流策略。
- 稳定性：超时、429、5xx 比例；重试后成功率。
- 工具链：评测脚本、灰度切流、A/B 实验可行性。

### 3.5 风险合规审查
- 数据合规：传输、存储、日志脱敏。
- 输出安全：有害内容、幻觉风险、可控性。
- 许可条款：商用限制、地域合规要求。

## 4. 分阶段执行策略（建议）

### Phase 3（当前）
目标：保障“录入 -> 内化 -> 检索向量”主链路稳定。
- 先评：parse/internalize/embedding
- 指标最小集：质量、P95 时延、单位成本、失败率
- 输出：Phase3 模型选型表 v0 + 默认路由建议

### Phase 4
目标：图谱关系质量可控。
- 新增评测：relation_type 判型准确率、证据摘要可用性
- 输出：图谱任务独立模型建议（可与 QA/内化分离）

### Phase 5
目标：QA 检索决策稳定。
- 新增评测：意图分类稳定性、扩图收益、过检索率
- 输出：决策模型选型与停止规则参数

### Phase 6
目标：回答质量与可信边界达标。
- 新增评测：答案忠实性、自然度、边界表达一致性
- 输出：回答模型最终路由与主备方案

## 5. 选型产物清单（每轮都要产出）

每一轮评测必须产出：
1. 候选模型列表（国内/国外）
2. 任务-模型打分表
3. 默认模型 + 回退模型建议
4. 风险与不适用场景
5. 下一轮待验证假设

## 6. 打分模型（可直接用）

建议总分 = 质量 45% + 时延 20% + 成本 20% + 稳定性 15%

- 质量：任务准确率、格式遵循率、证据忠实性
- 时延：P50/P95
- 成本：单请求平均成本（含重试）
- 稳定性：失败率、超时率、重试后成功率

合规不计分，作为“硬门槛”：不通过直接淘汰。

## 7. Prompt 调优路线

- Prompt v0：结构正确（先保证输出格式与字段稳定）
- Prompt v1：证据优先（事实/推断强分区，减少幻觉）
- Prompt v2：任务细分（摘要/标签/概念/判型分任务 prompt）
- Prompt v3：稳定性优化（对抗样本、长文本、边界场景）
- Prompt v4：成本优化（在质量达标前提下压 token 与时延）

## 8. 文档与数据迭代节奏

- 每周一次小迭代：补样本、更新分数、调整路由。
- 每个 Phase 结束做一次里程碑评审：确认是否切换默认模型。
- 保留历史版本：
  - v0（快速决策）
  - v1（补全安全/长文本/高并发）
  - v2（生产稳定后固化）

## 9. 下一步（立刻可执行）

1. 建立 Phase3 模型选型表 v0（parse/internalize/embedding）。
2. 选 10-20 条真实样本，覆盖短文本、长文本、附件、deferred。
3. 对每个候选模型跑同一批样本并记录：质量分、P95、成本、失败率。
4. 给出当前默认模型与回退模型建议，并写入环境变量路由策略。

## 10. 本地优先与部署预留口（当前约束）

当前决策：
- 先实现本地可用链路（开发阶段优先可验证）。
- 代码必须预留“生产部署可切换口”，避免未来重构主流程。

### 10.1 反爬抓取能力分层（必须遵守）
- 主业务链路：`capture -> parse -> internalize -> graph -> QA` 不直接耦合具体抓取实现。
- 抓取实现作为 provider 层：本地 provider 与线上 provider 可替换。
- 任何平台特定抓取（如小红书）都通过统一接口回传：
  - `content`（正文）
  - `image_urls` / `video_urls`
  - `notes`（诊断）
  - `platform`

### 10.2 本地阶段默认实现
- 默认使用本地抓取能力（静态解析 + 浏览器渲染兜底）。
- 小红书等反爬平台允许本地登录态会话参与抓取（仅开发环境）。
- 若命中登录墙/反爬，必须显式打诊断标记，不能伪装成功。

### 10.3 生产部署预留口（不在本轮实现）
- 通过环境变量切换外部抓取 provider，不改业务 API：
  - `EXTERNAL_XHS_CRAWLER_ENDPOINT`
  - `EXTERNAL_XHS_CRAWLER_TOKEN`
- 业务代码只依赖统一返回结构，不依赖某个开源项目内部字段。
- provider 异常时必须可降级，不允许阻断主流程（返回可解释失败 notes）。

### 10.4 验收标准（本地阶段）
- 本地可稳定复现抓取结果与诊断结果。
- 失败场景有明确可观测标记（如登录墙、无媒体地址、provider失败）。
- 切换到外部 provider 时，只改配置不改调用方代码。
