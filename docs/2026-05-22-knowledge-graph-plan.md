# INNEX 知识图谱模块实施计划（含后续保活优化）

日期：2026-05-22  
适用范围：`apps/web`（`/kb` 知识库模块）

## 1. 目标
- 在现有数据结构基础上，快速上线可用的知识图谱（MVP）。
- 不依赖 `background-base.html` 的星图样式，采用功能优先的图谱界面。
- 预留后续优化路径，保证可持续迭代（保活）。

## 2. 当前基础与可复用数据

### 2.1 可直接用于图谱的数据表
- `notes`：图谱节点主来源。
- `note_relations`：图谱边主来源。
- `note_chunks`：关系生成依据（后端增强用，不直接渲染）。

### 2.2 关键字段映射
- 节点（Node）：
  - `notes.id` -> `node.id`
  - `notes.title` -> `node.label`
  - `notes.summary` -> `node.summary`
  - `notes.concepts` / `notes.tags` -> 节点分组、筛选、着色依据
  - `notes.source` / `notes.source_url` -> 来源信息
  - `notes.capture_item_id` -> 收录箱定位联动
- 边（Edge）：
  - `note_relations.source_note_id` -> `edge.source`
  - `note_relations.target_note_id` -> `edge.target`
  - `note_relations.relation_type` -> `edge.type`

## 3. 方案原则
- 先上线：先用已存在数据完成图谱可视化闭环。
- 低侵入：优先新增接口与前端组件，不大改已有流程。
- 可演进：字段增强与算法增强按阶段推进。

## 4. 分阶段实施计划

## Phase A（MVP）图谱上线
目标：在 `/kb` 页面看到真实图谱并可基础交互。

工作项：
1. 新增 `GET /api/graph`
- 返回当前用户的 `nodes + edges`。
- 节点来自 `notes`，边来自 `note_relations`。
- 支持基础筛选参数：`search`、`tag`、`concept`（可选）。

2. 改造 `/kb` 页面为图谱视图（G6）
- 渲染节点与边。
- 支持交互：拖拽、缩放、点击节点、悬停高亮邻接节点。
- 点击节点后右侧展示笔记详情（标题、摘要、标签、关联）。

3. 联动能力
- 支持 `captureItemId` 入参定位并高亮对应节点。
- 支持从图谱详情切到关联节点。

验收标准：
- 有数据时可稳定渲染图谱。
- 50~300 节点内交互流畅。
- 收录箱“知识库定位”可正确定位到图谱节点。

## Phase B（关系质量优化）
目标：边更可信、图谱可解释性增强。

工作项：
1. `note_relations` 字段增强
- 新增 `confidence`（0~1）
- 新增 `is_auto_generated`（boolean）
- 新增 `evidence`（text/jsonb，记录关系依据）

2. 内化流程增强
- `/api/internalize` 在生成关系时写入 `confidence/evidence`。
- 前端支持关系阈值过滤（例如只看 `confidence >= 0.75`）。

验收标准：
- 用户可区分自动关系与高置信关系。
- 同一主题下误连边明显减少。

## Phase C（规模与体验优化）
目标：支持更大图谱和更稳定交互。

工作项：
1. 性能策略
- 节点超阈值时分层加载（先主题聚合，再展开笔记）。
- 大图分页/局部加载（按中心节点 n-hop 展开）。

2. 图谱缓存
- 增加图谱快照或布局缓存（如 `graph_snapshots` / `graph_node_metrics`）。
- 保持用户二次打开时布局稳定，降低重排闪动。

3. 运营可观测
- 记录渲染规模、接口耗时、交互失败率。

验收标准：
- 1000 级节点场景仍可操作（通过分层/局部加载）。
- 重进页面布局稳定，首屏时间可控。

## 5. 数据库与接口建议（最小改动优先）

短期（MVP）：
- 不改库，直接复用 `notes + note_relations`。

中期（Phase B）：
- 对 `note_relations` 增加：
  - `confidence numeric(4,3)`
  - `is_auto_generated boolean default true`
  - `evidence jsonb default '{}'::jsonb`

长期（Phase C）：
- 视性能需求增加缓存表，不提前过度设计。

## 6. 风险与应对
- 风险1：关系类型单一（当前多为 `related`）
  - 应对：Phase B 引入 `confidence + evidence`，再逐步丰富 `relation_type`。
- 风险2：图谱规模增长导致卡顿
  - 应对：Phase C 分层加载与布局缓存。
- 风险3：LLM 关系误判
  - 应对：显示关系依据，并支持手动修正/屏蔽（后续项）。

## 9. Phase A 接口契约（`/api/graph`）

### 9.1 Endpoint
- `GET /api/graph`

### 9.2 Query 参数（可选）
- `search`: string  
  - 按节点标题/摘要模糊匹配。
- `tag`: string  
  - 按 `notes.tags` 包含筛选。
- `concept`: string  
  - 按 `notes.concepts` 包含筛选。
- `limit`: number  
  - 最大返回节点数，默认 `300`，上限建议 `1000`。
- `focusNoteId`: string  
  - 可选焦点节点 ID，返回时附带高亮建议。

### 9.3 成功响应（200）
```json
{
  "nodes": [
    {
      "id": "note_uuid",
      "label": "笔记标题",
      "summary": "摘要",
      "tags": ["RAG", "检索"],
      "concepts": ["混合检索", "重排"],
      "source": "Notion",
      "sourceUrl": "https://example.com",
      "captureItemId": "capture_uuid",
      "createdAt": "2026-05-22T10:00:00.000Z",
      "updatedAt": "2026-05-22T10:00:00.000Z",
      "degree": 3
    }
  ],
  "edges": [
    {
      "id": "relation_uuid",
      "source": "source_note_uuid",
      "target": "target_note_uuid",
      "type": "related",
      "createdAt": "2026-05-22T10:00:00.000Z"
    }
  ],
  "meta": {
    "nodeCount": 120,
    "edgeCount": 260,
    "truncated": false,
    "limit": 300,
    "focusNoteId": "optional_note_uuid"
  }
}
```

### 9.4 字段说明
- `nodes[].degree`: 后端可直接计算（节点关联边数量），前端用于节点大小。
- `meta.truncated`: 当命中数据超过 `limit` 时为 `true`，前端显示“已截断”提示。
- `focusNoteId`: 原样回传，前端据此做初始高亮与居中。

### 9.5 错误响应
- `401 Unauthorized`
```json
{ "error": "Unauthorized" }
```
- `500 Internal Server Error`
```json
{ "error": "xxx error message" }
```

### 9.6 兼容 Phase B 的前向预留（可选）
Phase A 可先不返回以下字段；后续可无破坏追加：
- `edges[].confidence`
- `edges[].isAutoGenerated`
- `edges[].evidence`
