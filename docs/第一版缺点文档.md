# 第一版缺点文档

## 1. 解析链路稳定性不足（URL/PDF）
- 问题：外链抓取偶发失败；PDF 在录入/内化阶段存在超时、空文本、OCR回退不稳定。
- 影响：录入成功但内容质量波动，内化阶段可能被阻塞。
- 评估指标：
1. `capture_parse_success_rate`（录入阶段可读解析成功率）目标 >= 95%
2. `internalize_parse_success_rate`（内化阶段拿到原文率）目标 >= 98%
3. `url_fetch_success_rate`（外链正文抓取成功率）目标 >= 90%
4. `p95_parse_latency` 目标：URL < 8s，PDF < 20s

## 2. 异步状态一致性风险较高
- 问题：内化流程存在多状态（列表、详情、草稿、后台异步），曾出现状态显示与真实执行不一致。
- 影响：用户误判“未执行/已完成”，重复操作增多。
- 评估指标：
1. `ui_state_mismatch_rate`（状态不一致事件/千次操作）目标 < 1
2. `internalize_session_resume_success`（关闭后重开恢复内化页）目标 = 100%
3. `false_idle_rate`（实际处理中但UI显示空闲）目标 = 0

## 3. 兜底策略较多，行为可预期性不足
- 问题：历史上出现过“无原文也生成通用笔记”“占位文案被当正文”等兜底泄漏。
- 影响：数据可信度下降，用户需要额外人工核验。
- 评估指标：
1. `fallback_generation_rate`（兜底生成占比）目标 < 2%
2. `no_source_but_success_rate`（无原文却内化成功）目标 = 0
3. `placeholder_leak_rate`（占位文案落库率）目标 = 0

## 4. 标签/摘要质量稳定性不足
- 问题：标签模型超时、空标签（如 `-`）、摘要与标签补全时机不一致。
- 影响：检索与列表筛选效果下降。
- 评估指标：
1. `valid_tag_coverage`（至少1个有效标签占比）目标 >= 98%
2. `summary_nonempty_rate`（摘要非空占比）目标 >= 99%
3. `tag_timeout_rate` 目标 < 3%
4. `manual_edit_after_internalize_rate`（内化后大改率）应逐月下降

## 5. 前端表格与滚动体验韧性一般
- 问题：表头字段错位、滚动条可见性差、滚动行为需多次修正。
- 影响：可读性和操作效率受损。
- 评估指标：
1. `layout_regression_count`（每版本UI布局回归数）目标 < 2
2. `scroll_usability_pass_rate`（滚动可用性测试通过率）目标 = 100%
3. `cross_browser_ui_pass_rate`（Chrome/Edge/Safari）目标 >= 95%

## 6. 可观测性有基础但运营化不足
- 问题：已有日志，但缺少统一看板、自动告警、固定口径报表。
- 影响：定位依赖人工，修复效率受限。
- 评估指标：
1. `trace_coverage`（关键链路 trace_id 覆盖率）目标 = 100%
2. 告警覆盖：解析失败率、超时率、内化失败率、状态不一致率（4类）
3. `MTTR`（P1平均修复时间）目标 < 30 分钟

## 7. 交互性能仍有提升空间
- 问题：部分场景反馈感知慢，用户容易误触发重复操作。
- 影响：体感卡顿，操作信心下降。
- 评估指标：
1. 列表 `TTI`（可交互时间）目标 < 1.5s
2. `interaction_to_feedback`（点击到状态反馈）目标 < 150ms
3. `internalize_click_to_progress`（点保存到“处理中”可见）目标 < 100ms

## 建议的第一版验收标准
1. 解析成功率：URL >= 90%，内化拿原文率 >= 98%
2. 数据可信：无原文不入库（通用兜底率接近0）
3. 标签摘要：有效标签覆盖 >= 98%，摘要非空 >= 99%
4. 状态一致：内化会话恢复成功率 100%，状态错位事件/千次 < 1
5. 体验性能：列表TTI < 1.5s，点击反馈 < 150ms

## A. 录入解析（Capture Parse）实现总结

### 当前实现
1. 入口：`/api/capture-items`，支持文本、URL、图片、文档上传。
2. 类型识别：先做 `detectType`，再按来源选择解析输入（`url_content`/文件抽取文本/文本内容）。
3. URL：前端预抓取 `parse-url`，后端可二次 `refetch`；抓不到正文时不再把 URL 字符串当正文。
4. 文档：录入阶段有大小阈值；小文件尝试抽取，超阈值可跳过到内化阶段再处理。
5. 生成：调用 LLM 产出标题/摘要/标签，标签失败时有启发式 fallback。
6. 结果记录：写入 `parse_debug`（input_source、notes、错误原因）。

### 主要优点
1. 有清晰的分层策略（检测 -> 抽取 -> 生成 -> 回写）。
2. 有 fallback，避免“完全不可用”。
3. 有 `parse_debug` 可追踪问题。

### 主要缺点
1. URL 抓取受站点反爬、鉴权、网络波动影响明显。
2. 模型超时仍会影响标签质量一致性。
3. 阈值与策略较多，用户心智成本高。

### 对比指标（优化前后）
1. URL 抓取成功率（`url_fetch_success_rate`）
2. 录入解析成功率（`capture_parse_success_rate`）
3. 标签有效率（`valid_tag_coverage`）
4. 录入解析 P95 时延（`p95_parse_latency_capture`）

## B. 内化（Internalize）实现总结

### 当前实现
1. 两阶段：`dry_run=true` 生成草稿，`dry_run=false` 确认入库。
2. 强约束：拿不到原文直接 422 阻断，不允许“通用知识兜底笔记”入库。
3. 原文来源：`raw_content + 附件抽取文本`，并做最小文本长度校验。
4. 附件抽取：优先 parser-service，再 URL 读取，失败走 Supabase Storage 下载兜底。
5. 入库：写 `notes`、`note_chunks`、关系推断，并更新 `capture_items` 状态/摘要/标签。
6. UI：支持内化会话持续、关闭后恢复、保存后后台完成。

### 主要优点
1. 数据可信度显著提升（无原文不内化）。
2. 草稿可编辑，符合“人机共创”流程。
3. 会话状态管理已支持恢复。

### 主要缺点
1. 异步流程复杂，前后端状态一致性维护成本高。
2. 解析耗时长时，用户感知“卡住”风险仍存在。
3. 依赖多组件（parser-service/存储/模型），故障面较大。

### 对比指标（优化前后）
1. 原文可用率（`internalize_parse_success_rate`）
2. 内化成功率（`internalize_success_rate`）
3. 内化会话恢复成功率（`internalize_session_resume_success`）
4. 点保存到反馈时延（`internalize_click_to_progress`）

## C. 知识图谱（KB Graph）实现总结

### 当前实现
1. 数据接口：`/api/graph` 提供节点、边、统计信息。
2. 前端渲染：G6 力导布局，支持筛选、局部展开、关系批量处理。
3. 交互：节点选中高亮、邻域聚焦、缩放与导出。
4. 稳定性增强：图实例操作加了安全守卫与异常兜底，避免未就绪时报错刷屏。

### 主要优点
1. 功能覆盖较全（展示、筛选、批处理、导出）。
2. 关系置信度可视化，支持人工治理。

### 主要缺点
1. 大图性能与交互稳定性仍取决于数据规模和浏览器资源。
2. 图谱质量强依赖上游解析与关系抽取质量。

### 对比指标（优化前后）
1. 图谱首屏渲染时间（`graph_first_render_ms`）
2. 图交互异常率（`graph_runtime_error_rate`）
3. 关系人工修正率（`relation_manual_fix_rate`）

## D. QA（问答）实现总结

### 当前实现
1. 接口：`/api/qa`，会话化提问。
2. 检索：结合 chunk/向量（及 fallback 检索）召回候选证据。
3. 生成：基于证据回答，强调“证据不足时不强答”。
4. 结果：可在详情页保存 QA 记录，支持会话追问。

### 主要优点
1. 已有会话上下文与证据导向框架。
2. 与内化笔记联动，形成“收录-内化-问答”闭环。

### 主要缺点
1. 回答质量受召回质量和 chunk 质量影响大。
2. 缺少统一的离线评测集与自动回归评测体系。

### 对比指标（优化前后）
1. 首答命中率（`qa_first_answer_hit_rate`）
2. 证据充分率（`qa_evidence_sufficiency_rate`）
3. 无依据幻觉率（`qa_hallucination_rate`）
4. QA 平均响应时间（`qa_avg_latency_ms`）
