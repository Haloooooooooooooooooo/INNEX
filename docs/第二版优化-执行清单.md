# 第二版优化-执行清单

## 使用说明

- 本文档是基于 [第二版优化.md](/e:/my_vibecoding/INNEX/docs/第二版优化.md) 展开的可执行任务清单。
- 当前全部任务均视为未完成，统一使用 `- [ ]` 标记。
- 实施时应严格按 Phase 顺序推进，不建议跳阶段并行大改。
- 每个阶段完成后，应先做阶段验收，再进入下一阶段。

## Phase 1：数据与资产边界整理

### 1.1 资产与对象职责对齐

- [ ] 明确 `capture_items` 的正式职责说明，固定为：收录入口、原文快照入口、流转状态、轻解析状态。
- [ ] 明确 `notes` 的正式职责说明，固定为：正式内化资产。
- [ ] 明确 `note_relations` 的正式职责说明，固定为：正式图谱资产。
- [ ] 明确 `note_chunks` 的正式职责说明，固定为：内化检索中间资产。
- [ ] 明确 QA 相关表的职责说明，固定为：长期交互结果资产，而非知识本体资产。

### 1.2 `capture_items` 顶层字段重新定性

- [ ] 盘点当前 `capture_items` 现有字段，并按“身份字段 / 原文视图字段 / 流转字段 / 状态字段”分类。
- [ ] 固定 `id/user_id/type/title/source/source_url/my_understanding/status/created_at/updated_at` 为顶层长期保留字段。
- [ ] 固定 `raw_content` 的定位为“聚合主视图”，不是唯一 canonical 原文。
- [ ] 固定 `summary/tags` 在 `capture_items` 上的定位为 capture 阶段过渡资产。
- [ ] 固定 `parse_debug` 在 `capture_items` 上的定位为轻量解析状态与诊断资产。

### 1.3 `capture_items` 与 `notes` 关系重述

- [ ] 明确当前产品心智仍采用“一条 capture 对应一篇主 note”。
- [ ] 明确实现层不要把该关系写死成永远不可演化的一对一。
- [ ] 明确 `raw_content` 与 `note.content` 的职责区分：
  - `raw_content` 为材料层
  - `note.content` 为理解层
- [ ] 明确 `capture_items.summary/tags` 与 `notes.summary/tags/concepts` 的优先级关系：
  - capture 为过渡版本
  - note 为正式版本

### 1.4 资产持久化分层

- [ ] 固定长期资产清单：原文资产、正式内化资产、正式图谱资产、关键 QA 结果资产。
- [ ] 固定中间资产清单：切片、embeddings、补解析结果、事实层片段。
- [ ] 固定运行时资产清单：检索候选集、扩图路径、临时 agent 判断、临时上下文拼接。
- [ ] 明确各现有对象分别属于哪一层资产，不允许后续实现时继续混用。

### 1.5 Phase 1 验收

- [ ] 输出一份数据对象职责映射表。
- [ ] 输出一份字段分层说明表。
- [ ] 确认后续 Phase 设计不再把 `capture_items` 当作“万能下游容器”。

## Phase 2：原文资产与切片基础设施补强

### 2.1 原文来源体系定义

- [ ] 定义原文来源类型最小集合：
  - 用户输入文本
  - URL 正文
  - 附件提取文本
  - OCR / 图片补充文本
  - 其他补充原始文本
  - 用户理解
- [ ] 明确哪些来源属于“主原文”，哪些来源属于“补充原文”。
- [ ] 明确用户理解不并入客观原文主体。
- [ ] 明确原文资产需要保留来源边界与最小元信息。

### 2.2 原文来源结构落位

- [ ] 在不推翻现有 `capture_items` 主骨架的前提下，设计来源级原文承载方式。
- [ ] 保证来源级原文可以追溯到 URL、附件或其他原始来源标识。
- [ ] 保证来源级原文可以标记是否为主正文。
- [ ] 保证来源级原文可以标记解析是否成功、是否 deferred。
- [ ] 保证 `raw_content` 可由来源级原文聚合生成，但不替代来源级原文本身。

### 2.3 双层切片体系

- [ ] 固定“内化切片”和“原文切片”双体系并存，不混为一套。
- [ ] 规定内化切片的目标为知识定位、主题检索、图谱关系粗筛。
- [ ] 规定原文切片的目标为证据支撑、引用回溯、事实校验。
- [ ] 规定原文切片要保留更强的上下文连续性。
- [ ] 规定内化切片要更贴近语义结构与知识单元。

### 2.4 原文资产与内化层衔接

- [ ] 明确内化阶段如何优先复用 capture 阶段已保存的原文资产。
- [ ] 明确哪些情况必须回到来源级原文，而不是只读 `raw_content`。
- [ ] 明确哪些补解析结果要沉淀为原文资产，而不是只作为一次性流程结果。

### 2.5 Phase 2 验收

- [ ] 输出原文来源分类规范。
- [ ] 输出原文聚合规则。
- [ ] 输出双层切片职责说明。
- [ ] 确认后续图谱和 QA 设计都能“回到原文”。

## Phase 3：内化层升级

### 3.1 内化职责重构

- [ ] 明确内化不再只是“生成一篇笔记”。
- [ ] 固定内化职责为：
  - 补解析 deferred / 大文件内容
  - 生成结构化正式笔记
  - 生成正式摘要 / 标签 / 概念
  - 产出图谱和 QA 可用的正式资产

### 3.2 内化输入规范

- [ ] 固定内化输入三层：
  - 原始正文主体
  - 来源补充内容
  - 用户理解引导
- [ ] 明确 `raw_content` 是默认主输入，不是唯一输入。
- [ ] 明确大文件 / deferred 内容在内化阶段如何补齐。
- [ ] 明确视频解析不作为当前前提，后续可作为补充来源接入。

### 3.3 内化模型与 prompt 设计方向

- [ ] 固定内化模型优先级：
  - 忠于原文
  - 有洞察和关联
- [ ] 明确内化 prompt 不能只要求摘要压缩。
- [ ] 明确内化 prompt 必须强制区分：
  - 原文明确支持内容
  - 推断 / 延展内容
- [ ] 明确 capture 阶段模型策略与 internalize 阶段模型策略允许分离。

### 3.4 内化正文结构固化

- [ ] 固定内化正文必备板块：
  - 核心内容
  - 关键概念 / 关键信息
  - 原文支持要点
- [ ] 固定内化正文强烈建议板块：
  - 推断与延展
  - 可关联方向
- [ ] 明确“原文支持要点”和“推断与延展”在展示上必须让用户一眼区分。

### 3.5 内化正式资产产出

- [ ] 固定正式资产清单：
  - 正式笔记正文
  - 正式摘要
  - 正式标签
  - 正式概念
  - 事实层片段
- [ ] 明确推断层片段当前至少在正文中分区，后续再决定是否单独资产化。
- [ ] 明确正式摘要 / 标签 / 概念以 note 层版本为主，不再以 capture 层版本为主。

### 3.6 Phase 3 验收

- [ ] 输出内化输入规范。
- [ ] 输出内化正文结构规范。
- [ ] 输出内化正式资产清单。
- [ ] 确认内化结果既忠于原文，又能为图谱和 QA 提供正式入口。

## Phase 4：图谱层增强

### 4.1 图谱节点与关系范围

- [ ] 固定当前图谱主节点为 `notes`。
- [ ] 明确当前阶段不引入完整实体主节点体系。
- [ ] 固定当前最小稳定关系集：
  - `related`
  - `supports`
  - `example_of`
- [ ] 明确当前阶段不引入 `contradicts`。

### 4.2 图谱关系生成流程

- [ ] 固定关系生成三段式：
  - 粗筛召回
  - 模型判型
  - 证据落库
- [ ] 固定粗筛信号优先级：
  - embedding similarity
  - tags / concepts overlap
  - title / summary / keyword overlap
- [ ] 固定判型主输入：
  - 正式摘要
  - 正式标签 / 概念
  - 事实层片段
- [ ] 固定判型辅助输入：
  - 核心正文
  - 原文片段
- [ ] 明确 `my_understanding` 和推断层内容不作为图谱边主判定依据。

### 4.3 图谱边证据体系

- [ ] 为每条边补齐 `relation_type`。
- [ ] 为每条边补齐 `confidence`。
- [ ] 为每条边补齐 `evidence_summary`。
- [ ] 明确置信度至少具备分层意义，不要求一开始追求非常精密数值。
- [ ] 明确证据摘要优先引用共享概念、事实层片段、原文支持片段和简短模型理由。

### 4.4 图谱与原文 / 内化资产分工

- [ ] 固定“关系发现主要依赖内化资产”。
- [ ] 固定“关系证据尽量回到原文资产”。
- [ ] 明确图谱不是事实层替代品，而是关系导航层。

### 4.5 Phase 4 验收

- [ ] 输出图谱关系类型规范。
- [ ] 输出图谱关系生成规则。
- [ ] 输出图谱边证据与置信度规则。
- [ ] 确认图谱边对 QA 有可直接利用的结构价值。

## Phase 5：QA 检索与 Agent 决策升级

### 5.1 QA 总体形态固定

- [ ] 固定 QA 为 agent-guided RAG。
- [ ] 明确 QA 不等同于普通单轮向量问答。
- [ ] 明确 QA 也不等同于无检索结构的自由 agent。

### 5.2 QA 检索顺序

- [ ] 固定检索顺序：
  - 先查内化资产
  - 再补原文资产
  - 不够再扩图
- [ ] 明确图谱默认承担第二阶段扩检索职责，而不是首跳入口职责。
- [ ] 明确内化切片负责知识定位，原文切片负责证据支撑。

### 5.3 QA 意图分类

- [ ] 固定主问题意图分类：
  - fact query
  - concept explanation
  - summary / synthesis
  - comparison / distinction
  - action advice
- [ ] 固定扩展型意图分类：
  - evidence strengthening
  - example / case request
  - related-topic expansion
- [ ] 明确 `supports / example_of / related` 属于图谱扩检索偏好，而不是完整 QA 意图分类本身。

### 5.4 多轮追问与语义连续性判断

- [ ] 明确“连续消息”不等于“相关追问”。
- [ ] 建立语义连续性判断信号：
  - 指代表达
  - 主题延续
  - 追问类型
  - 与上一轮命中资产的适配度
- [ ] 明确只有在语义连续时，才允许提前借助图谱扩检索。

### 5.5 图谱扩检索决策

- [ ] 固定追问扩图偏好：
  - evidence strengthening -> `supports`
  - example request -> `example_of`
  - related-topic expansion -> `related`
- [ ] 明确判断不清时必须保守，不激进扩图。
- [ ] 明确图谱扩检索不是越多越好，而是为当前问题补足缺失上下文。

### 5.6 Agent 停止规则

- [ ] 固定“够用即停”原则。
- [ ] 明确停止条件：
  - 证据已足够
  - 扩展收益下降
  - 继续扩也无法补齐关键证据
- [ ] 明确回答策略：
  - 能答 -> 直接答
  - 能部分答 -> 带边界答
  - 不能可靠答 -> 明确证据不足

### 5.7 Phase 5 验收

- [ ] 输出 QA 检索顺序说明。
- [ ] 输出 QA 意图分类规范。
- [ ] 输出多轮追问判断规则。
- [ ] 输出图谱扩检索与停止规则。

## Phase 6：回答生成与 QA 模型 / Prompt 优化

### 6.1 QA 双角色心智

- [ ] 固定 QA 双角色：
  - 决策角色
  - 回答角色
- [ ] 明确当前阶段允许同一模型承担两种角色，但心智与 prompt 必须分开。

### 6.2 决策角色优化目标

- [ ] 固定决策角色优先级：
  - 稳定分类
  - 保守判断
  - 扩图决策
  - 停止决策
- [ ] 明确决策角色不追求“会聊天”，只追求稳定可执行判断。

### 6.3 回答角色优化目标

- [ ] 固定回答角色优先级：
  - 忠于证据
  - 多源上下文整合
  - 自然解释
  - 边界 / 不确定性表达
- [ ] 明确回答模型优先追求“证据忠实地整合并自然表达”，而不是“显得很聪明”。

### 6.4 决策 prompt

- [ ] 设计决策 prompt 只做路由，不做长答。
- [ ] 决策 prompt 固定判断项：
  - QA 意图
  - 是否语义连续
  - 是否扩图
  - 优先关系类型
  - 是否保守处理
- [ ] 决策 prompt 设计原则固定为：
  - 短
  - 硬
  - 结构化
  - 非聊天化

### 6.5 回答 prompt

- [ ] 设计回答 prompt 先回答、后说明依据，不暴露检索决策过程。
- [ ] 固定回答 prompt 必须做到：
  - 以证据为边界
  - 重组，不拼贴
  - 自然解释
  - 体现依据
  - 体现边界
- [ ] 固定回答结构骨架：
  - 结论
  - 依据
  - 关联补充
  - 边界 / 不确定性
- [ ] 明确该结构是内在骨架，不要求表面机械模板化。

### 6.6 回答风格与不确定性表达

- [ ] 固定“回答要像人说的话”作为产品级要求。
- [ ] 明确回答不能是检索片段拼接。
- [ ] 固定不确定性表达方式：
  - 先给当前最稳判断
  - 再说明边界或证据不足
- [ ] 明确不确定性以后置边界说明出现，不以前置弱化语气主导整段回答。

### 6.7 Phase 6 验收

- [ ] 输出决策 prompt 规范。
- [ ] 输出回答 prompt 规范。
- [ ] 输出回答结构与风格规范。
- [ ] 确认回答生成层符合“自然、可信、带边界感”的目标。
