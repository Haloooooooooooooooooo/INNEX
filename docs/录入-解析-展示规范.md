# 录入 · 解析 · 展示规范

> 本文档为 INNEX 系统的核心交互规范，所有阶段（Phase 1/2/3）的录入、解析、收录箱展示、详情抽屉功能必须遵循本文档。

---

## 一、提交条件

**内容输入框有内容 或 附件区有文件 → 即可提交。** 两者都有也行，至少一个就行。两者皆空时提交按钮不可用。

---

## 二、类型判断（系统自动，用户不选）

```
内容框有 URL → 看域名：
  bilibili.com / b23.tv / youtube.com / youtu.be → type = "video"
  其他 URL                                        → type = "url"

内容框纯文字（无 URL）                              → type = "text"

内容框为空、只有附件：
  1 个图片文件  → type = "image"
  1 个文档文件  → type = "document"
  多个附件      → type = "attachment_group"
```

---

## 三、可读性判断

可读性决定标题、摘要、标签的生成方式。

| 类型 | 条件 | 可读？ | 说明 |
|------|------|--------|------|
| text | < 50 字 | ❌ | 文字太短，不生成摘要 |
| text | ≥ 50 字 | ✅ | 直接读取全文 |
| url（图文） | — | ✅ | fetch 页面，读标题和正文 |
| url（视频） | — | ⚠️ 半可读 | 读标题和附带文字，**不读视频内容本身** |
| document | 文件小 | ✅ | 调 Docling 解析 |
| document | 文件大 | ❌ | 暂不解析 |
| image | ≤ 3 张 | ✅ | OCR 识别 |
| image | > 3 张 | ❌ | 暂不解析 |
| attachment_group | 总大小 < 阈值 | ✅ | 逐个按类型处理 |
| attachment_group | 总大小 ≥ 阈值 | ❌ | 一个都不读 |

---

## 四、字段生成规则

### 4.1 标题

| 类型 | 可读/半可读 | 不可读 |
|------|------------|--------|
| text | 通过 LLM 从内容提炼标题 | 取前5字 |
| url（视频） | 直接从网页 `<title>` 提取 | —（总能 fetch） |
| url（图文） | 直接从网页 `<title>` 提取 | —（总能 fetch） |
| document | 文档文件名 | 文档文件名 |
| image | 通过 LLM 从 OCR 内容提炼 | `收录时间+图片` |
| attachment_group（1个） | 按对应类型规则 | 按对应类型规则 |
| attachment_group（多个） | — | `收录时间+附件组` |

### 4.2 来源

| 类型 | 来源值 |
|------|--------|
| text | `文字` |
| url · B站 | `B站` |
| url · YouTube | `YouTube` |
| url · 微信公众号 | `微信公众号` |
| url · 知乎 | `知乎` |
| url · 小红书 | `小红书` |
| url · 其他 | `链接` |
| document | `文档` + 类型（如 `文档PDF`、`文档Word`） |
| image | `图片` |
| attachment_group（1个） | 按对应类型 |
| attachment_group（多个） | `附件组` |

### 4.3 摘要

通过 LLM（DeepSeek）生成。

| 可读性 | 摘要值 |
|--------|--------|
| ✅ 可读 | LLM 根据读取内容生成摘要 |
| ⚠️ 视频（半可读） | LLM 根据文字部分生成摘要 |
| ❌ 文字 < 50 字 | 留空 |
| ❌ 文档过大 | `文件过大，内化后才能生成` |
| ❌ 图片过多 | `图片太多，内化后才能读取` |
| ❌ 附件过大 | `附件太大，内化后才能读取` |

### 4.4 标签

通过 LLM（DeepSeek）生成。

| 可读性 | 标签值 |
|--------|--------|
| ✅ 可读 | LLM 提取 3-8 个标签（字符串数组） |
| ❌ 不可读 | `["-"]`，等内化时再打标签 |

### 4.9 轻解析的 LLM 调用

可读内容在录入后会**同步调用 LLM** 生成标题、摘要、标签。提交按钮显示"正在解析…"，LLM 返回后记录完整入库。

**LLM 承担的部分：**

| 字段 | 条件 | 做法 |
|------|------|------|
| 标题 | type=text | LLM 从内容提炼标题 |
| 标题 | type=image（已 OCR） | LLM 从 OCR 文字提炼标题 |
| 摘要 | 所有可读类型 | LLM 生成 1-2 句中文摘要 |
| 标签 | 所有可读类型 | LLM 提取 3-8 个关键概念标签，返回 JSON 数组 |

**规则承担的部分（不走 LLM）：**

| 字段 | 做法 |
|------|------|
| 标题（URL） | `fetch` → 提取 `<title>` / `og:title` |
| 标题（文档） | 取文件名 |
| 来源 | 类型→来源映射表 |
| 可读性 | 字数/文件大小/图片数量比较 |
| 状态 | 用户点击的按钮 |

### 4.5 状态

| 用户点击 | 入库值 |
|----------|--------|
| 稍后看 | `later` |
| 收藏 | `pending` |

### 4.6 收录时间

格式：`YYYY-MM-DD HH:mm`（如 `2026-05-20 14:23`）

### 4.7 我的理解

用户有输入→保存原样；没有→留空。**不受可读性影响。**

### 4.8 附件

展示附件文件清单（文件名、类型、大小）。录入时不上传解析，仅记录元数据。

## 五、收录箱展示

### 5.1 Tab 筛选

| Tab | 筛选条件 |
|-----|----------|
| 全部 | 所有记录 |
| 稍后看 | `status = later` |
| 待内化 | `status = pending` |
| 已沉淀 | `status = crystallized` |

每个 Tab 显示对应记录数。

### 5.2 表格列（7 列）

```
☑ | 标题 | 来源 | 收录时间 | 状态 | 标签 | 操作
```

### 5.3 操作按钮（按状态）

| 状态 | 操作按钮 |
|------|----------|
| 稍后看 | 查看原笔记 · 转待内化 · 一键内化 · 删除 |
| 待内化 | 查看原笔记 · 一键内化 · 删除 |
| 已沉淀 | 查看原笔记 · 基于此笔记提问 · 知识库定位 · 删除 |

### 5.4 查看原笔记行为

| 类型 | 行为 |
|------|------|
| 链接（有 source_url） | `window.open` 跳转原网页 |
| 文字（无 source_url） | 弹窗展示文字内容（有关闭按钮） |
| 文档 | 下载/打开文件 |

## 六、详情抽屉

### 6.1 通用展示字段（所有状态）

标题 · 来源 · 收录时间（mono 字体）· 状态 · 标签 · 摘要 · 我的理解 · 附件清单

### 6.2 笔记本

所有状态的抽屉内都有一个**独立于"我的理解"的笔记本区域**。用户可以在这里随时记录与当前条目相关的笔记。这是一个自由文本区域，自动保存。

### 6.3 按状态的底部按钮

| 状态 | 按钮 |
|------|------|
| 稍后看 | `查看原笔记` `转待内化` `一键内化` `删除` |
| 待内化 | `查看原笔记` `一键内化` `删除` |
| 已沉淀 | `查看原笔记` `基于此笔记提问` `知识库定位` `删除` |

### 6.4 已沉淀额外区域

- **AI 笔记框**：展示内化生成的 Markdown 笔记，点击跳转完整笔记详情
- **AI 助手回答栏**：仅当有保存过的问答记录时才显示，列出历史问答（问题+答案摘要+来源引用）

---

## 七、一键内化草稿视图

点击"一键内化"后，抽屉内容切换为草稿模式：

1. 顶部：`内化草稿` 标题 + 提示文字
2. 主体：AI 笔记正文（可编辑的大文本框，预填 LLM 生成的结构化 Markdown）
3. 底部：
   - `返回详情` → 回到上一节的详情视图
   - `保存` → 笔记入库 → 状态变为 `crystallized`

---

## 八、删除规则

| 状态 | 删除行为 |
|------|----------|
| 稍后看 / 待内化 | 删除记录 + 附件引用 |
| 已沉淀 | 删除记录 + 对应笔记 + 关联关系 + AI 回答 |

所有删除操作需要二次确认弹窗。
