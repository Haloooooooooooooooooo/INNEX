# 第二版优化 Benchmark v0

## 文档目的

这份文档用于沉淀第二版优化的第一批真实样本，作为后续调优的统一 benchmark。

目标不是做学术评测，而是解决这几个实际问题：

- 每次改动后，我们能不能用同一批样本复跑
- 录入、解析、内化、图谱、QA 到底有没有变好
- 哪一类样本最容易出问题
- 后续应该优先优化哪一层

---

## Benchmark 使用原则

1. 这批样本优先服务当前系统的增量优化，不追求大而全。
2. 每条样本都尽量绑定“最需要验收的能力”，避免泛测。
3. 每次优化后，只要影响到某一层，就优先复跑相关样本。
4. 评估方式先用三档：
- `通过`
- `部分通过`
- `失败`

---

## 建议评估层

这批样本后续建议按 4 层复跑：

1. `录入解析`
- 类型识别是否正确
- 原文是否尽量保留
- summary 是否短、准、干净
- tags 是否像主题词而不是结构词

2. `内化`
- 是否忠于原文
- 是否区分事实层和推断层
- 是否把图片/文档/原文中的关键信息真正吃进去

3. `图谱`
- 该连的有没有连上
- 关系方向是否合理
- 是否能为后续 QA 提供有价值的关联

4. `QA`
- 是否答得对
- 是否引用/回到原文
- 是否在多轮场景里判断得合理

## 单条样本设计

## S01 小红书链接 01

- 类型：`url-xiaohongshu`
- 输入：`http://xhslink.com/o/4xmw4Y4tmBD`
- 核心测试目标：
  - 是否识别为小红书链接
  - 是否能拿到页面文字
  - 是否能统计图片数量
  - 是否按 `<=5 全量 OCR / >5 deferred` 规则分流
- 录入通过标准：
  - 类型识别正确
  - `parse_debug` 中能看到平台与图片分流信息
  - 如果图片不多，summary 不应只读页面介绍文字
  - 如果图片很多，录入应成功且标记 deferred
- 内化通过标准：
  - deferred 场景下，内化应补全图片 OCR
- 风险点：
  - 小红书反爬
  - 页面图片 URL 抽取不稳定
  - 多图导致录入过慢

## S02 Prompt A/B Test 方法论文本

- 类型：`text`
- 输入特征：
  - 长文本
  - 有强结构分段
  - 附带 `my_understanding`
- 我的理解：`Prompt的AB test该怎么做`
- 核心测试目标：
  - 长文本 summary 是否能压成简洁结论
  - tags 是否能提炼成主题词而不是段落词
  - `my_understanding` 是否被保留但不污染原文
- 录入通过标准：
  - summary 不超过 1-2 句
  - tags 不出现“第一步/第二步/本质/最后一步”等结构词
  - parse 阶段无异常超长输出
- 内化通过标准：
  - 能产出清晰的方法论结构
  - 应与 S03 建立较强关联
