## 二、认证与数据隔离

### 2.1 认证方案

```
┌──────────────────────────────────────────────────────────────┐
│ Supabase Auth                                                  │
│                                                                │
│ 注册方式:  邮箱 + 密码                                          │
│            (后续可选加 GitHub / Google OAuth)                   │
│                                                                │
│ 流程:                                                          │
│   /login        登录页  →  Supabase signInWithPassword        │
│   /register     注册页  →  Supabase signUp + 创建 profile     │
│   切换账户       登录页  →  退出当前 → 换账号登录               │
│   退出登录       →  Supabase signOut → 重定向 /login          │
│                                                                │
│ 前端状态:                                                       │
│   Supabase client.auth.getSession() → 当前用户身份              │
│   AuthContext (React Context) → 全局共享 user 对象              │
│                                                                │
│ 后端保护:                                                       │
│   API Routes 读取 request cookies 中的 supabase-auth-token     │
│   Supabase server client.auth.getUser(token) → 验证身份         │
│   所有查询 WHERE user_id = authenticated_user.id               │
│                                                                │
│ 数据库保护:                                                     │
│   RLS Policy 在所有表上强制 user_id = auth.uid()               │
│   即使 API 层漏了，数据库层也兜底                                │
└──────────────────────────────────────────────────────────────┘
```

### 2.2 数据隔离机制（三层防御）

```
第1层: 前端路由守卫
  /inbox, /kb, /ai → middleware.ts 检查 session
  未登录 → 重定向 /login

第2层: API Routes 鉴权
  每个 API Route 先读 cookie 中的 token
  调用 supabase.auth.getUser(token) 获取 user_id
  所有 Supabase 查询 .eq('user_id', userId)

第3层: 数据库 RLS
  ALTER TABLE capture_items ENABLE ROW LEVEL SECURITY;
  CREATE POLICY "user sees own data"
    ON capture_items FOR ALL
    USING (auth.uid() = user_id);
```

### 2.3 认证状态机

```
┌──────────┐   登录成功    ┌──────────┐   退出登录    ┌──────────┐
│  登录页   │ ──────────▶ │  应用页    │ ──────────▶ │  登录页   │
│ /login   │              │ /inbox    │              │ /login   │
│           │ ◀────────── │           │              │           │
│           │  未登录访问   │           │              │           │
│           │   /inbox     │           │              │           │
└──────────┘              └──────────┘              └──────────┘
                                │
                         切换账户按钮
                                │
                                ▼
                          signOut() →
                          回到 /login
                          数据完全不可见
```

### 2.4 账户切换流程

```
用户在 /inbox 页面点击"切换账户"
  ↓
Supabase signOut()         → 清除本地 session
  ↓
Next.js router.push('/login')
  ↓
用户在新登录页输入另一个邮箱密码
  ↓
Supabase signInWithPassword → 新 session
  ↓
router.push('/inbox')
  ↓
所有数据自动变成新用户的数据（因为 API 查询用新 user_id）
```

### 2.5 退出登录时数据保护

```
退出登录 = signOut() + router.push('/login')
  ↓
┌────────────────────────────────────────────┐
│ 前端: session 清除，所有 in-memory state 销毁 │
│ API:  后续请求无 token → 401                │
│ DB:   RLS 阻止无 user_id 的查询             │
│       任何 API 调用都无法查到数据             │
│ UI:   只显示登录页，无任何用户数据            │
└────────────────────────────────────────────┘
```

## 三、数据流全链路

一条内容从进入系统到变成可复用知识资产，经过 4 个阶段：

```
阶段0: 认证检查 (所有操作的前提)
  middleware.ts 验证 JWT → API Route 读取 user_id → RLS 兜底

阶段1: 收录 (Capture)
  用户输入 → POST /api/capture-items → capture_items 表 (带 user_id)
  状态: later (稍后看) 或 pending (待内化)

阶段2: 轻解析 (Light Parse)
  收录后同步触发 → 固定规则 → 填充 title / source / summary / tags
  ┌─────────────────────────────────────────────────────┐
  │ 文字 <50 字  → 不生成摘要, 标题取前 5 字            │
  │ 文字 ≥50 字  → 直接读取, 生成摘要                   │
  │ 文档 <阈值   → 调 Docling 解析, 生成摘要             │
  │ 文档 ≥阈值   → 暂不解析, 标记"内化后生成"            │
  │ 视频链接     → 暂不解析, 标记"内化后生成"            │
  │ 图文链接     → 读取页面内容, 生成摘要                │
  │ 图片 ≤3 张   → 直接 OCR, 生成摘要                   │
  │ 图片 >3 张   → 暂不解析, 标记"内化后生成"            │
  │ 附件组       → 按总大小判断是否解析                  │
  └─────────────────────────────────────────────────────┘

阶段3: 内化 (Internalization)  ← 核心
  用户点"一键内化" → POST /api/internalize → Agent → 直接入库
  ┌──────────────────────────────────────────────────────┐
  │ Step 1: 读取 capture_item + attachments (带 user_id) │
  │ Step 2: 判断内容类型 → 必要时调用解析工具            │
  │ Step 3: LLM 生成结构化 Markdown 笔记                  │
  │ Step 4: 提取标签 / 概念 / 关键结论                   │
  │ Step 5: 向量检索相似旧笔记 → 生成关联关系            │
  │ Step 6: 写入 notes 表 + note_relations 表            │
  │ Step 7: 更新 capture_items.status = 'done'           │
  │ Step 8: 触发图谱更新 (Graph Agent)                   │
  └──────────────────────────────────────────────────────┘
  全自动完成，无需用户确认中间步骤。

阶段4: 沉淀 (Crystallized)
  笔记可被: 知识库 RAG 问答 / 图谱浏览
  笔记可回写: 问答有价值时 → 保存到 ai_answers 表 → 出现在笔记详情
```

## 五、数据库设计

### 5.1 用户表

```sql
-- Supabase 内置 auth.users 表，无需自建
-- 扩展用户 profile（public.profiles）
CREATE TABLE public.profiles (
  id         UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  username   TEXT,
  avatar_url TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 注册时自动创建 profile
CREATE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.profiles (id, username)
  VALUES (new.id, new.email);
  RETURN new;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

### 5.3 RLS 策略（以 capture_items 为例）

```sql
ALTER TABLE capture_items ENABLE ROW LEVEL SECURITY;

-- 用户只能看到自己的记录
CREATE POLICY "Users can view own items"
  ON capture_items FOR SELECT
  USING (auth.uid() = user_id);

-- 用户只能插入自己的记录
CREATE POLICY "Users can insert own items"
  ON capture_items FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- 用户只能更新自己的记录
CREATE POLICY "Users can update own items"
  ON capture_items FOR UPDATE
  USING (auth.uid() = user_id);

-- 用户只能删除自己的记录
CREATE POLICY "Users can delete own items"
  ON capture_items FOR DELETE
  USING (auth.uid() = user_id);
```

## 六、API 设计

所有 API 都需要认证（除了 auth 相关端点）。

### 认证

```
POST   /api/auth/register       注册 (email + password + username)
POST   /api/auth/login          登录 (email + password)
POST   /api/auth/logout         退出
GET    /api/auth/me             获取当前用户信息
```

### 收录箱 CRUD（第一阶段）

```
GET    /api/capture-items          列表查询 (?status=&search=&page=&limit=)
                                   返回 user_id = 当前用户的数据
POST   /api/capture-items          新增记录
GET    /api/capture-items/[id]     单条详情 (需 owner 校验)
PATCH  /api/capture-items/[id]     更新字段
DELETE /api/capture-items/[id]     删除 (级联删除附件和笔记)
```

### 内化（第二阶段，全自动）

```
POST   /api/internalize            触发内化 → Agent 生成笔记 → 直接入库
       Body: { capture_item_id }
       → 内部流程: 读取记录 → 解析 → LLM 生成笔记 → 写入 notes → 更新状态
```

### 知识库问答（第二阶段，纯 RAG）

```
POST   /api/qa                     知识库提问 → 检索 + 生成回答
POST   /api/qa/save                保存回答到笔记
```

### 知识图谱（第三阶段）

```
GET    /api/graph                  获取全量图谱数据 (nodes + edges)

```

## 八、模块边界与依赖方向

依赖方向是单向的：

```
Auth Layer (认证路由守卫)
         │
         ▼
Inbox CRUD → Light Parse → Internalization Agent → Notes Table
                                                         │
                                          ┌──────────────┼──────────────┐
                                          ▼                             ▼
                                     Knowledge QA              Knowledge Graph
                                      (纯 RAG 引擎)              (G6 + Graphin)
                                          │                             │
                                          └──────────────┬──────────────┘
                                                         ▼
                                                  ai_answers Table
                                                  (回写到笔记详情)
```

关键约束：
- **所有请求经过 Auth 层**。middleware.ts 做路由守卫，API Routes 做身份校验。
- **低层模块不依赖高层模块**。Inbox CRUD 不知道 Agent 存在。
- **数据通过数据库表通信，不是函数调用**。
- **每个用户的数据完全隔离**。前端、API、数据库三层保护。

## 九、渐进式交付路线

```
        现在         第一阶段             第二阶段            第三阶段
         │              │                    │                  │
    ─────┼──────────────┼────────────────────┼──────────────────┼─────▶
         │              │                    │                  │
   background-    认证系统 + 布局       内化 Agent        知识图谱
   base.html      + 收录箱 CRUD        + RAG 问答         + G6 + Graphin
   (原型参考)     + 设计 tokens        + Docling 解析     + 提醒反馈
                  + 状态流转           + 真正文件上传     + 悬停高亮联动
```

每一阶段都是一个**可跑的应用**。第一阶段完成后：
- 用户可以注册/登录/退出
- 可以录入内容、查看列表、切换状态
- 不同账户的数据完全隔离
- 退出登录后看不到任何数据

## 十、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 认证 | Supabase Auth (邮箱+密码) | 与数据库同源，RLS 天然打通 |
| 数据库 | Supabase PostgreSQL + pgvector | 一表存业务+向量，不引入独立向量库 |
| 行级安全 | RLS ON → auth.uid() = user_id | 数据库兜底，即使 API 漏了也安全 |
| LLM 选型 | DeepSeek（当前默认），后续按评估切换 | Phase 1 先用 DeepSeek，架构预留 provider 热切换 |
| LLM 多模态 | GPT-4o | DeepSeek 不支持时按需切换：图片 OCR/识别、视频理解 |
| LLM 调用层 | Vercel AI SDK（先），Mastra（后） | 第一阶段流程简单，后续复杂 tool calling 时升级 |
| 内化 Agent | 参考 LightRAG 的实体抽取 Prompt + 图谱构建思路 | 不引入依赖，借鉴设计，全自动（无需确认步骤） |
| RAG 检索 | 参考 LightRAG 的 local/global/mix 混合检索策略 | 三层检索：具体笔记 / 主题全貌 / 混合 |
| AI 问答 | 纯 RAG 引擎，非聊天机器人 | 只基于已沉淀笔记回答，证据不足明确说不确定 |
| 文档解析 | Docling FastAPI sidecar | Python 生态最强，HTTP 调用 |
| 文件存储 | Supabase Storage | 和数据库同源 |
| 知识图谱可视化 | G6 + Graphin（第三阶段引入） | 力导向布局 + Combo 分组 + Canvas 性能，替代 React Flow |
| Agent | 第二阶段引入 | 第一阶段无需 Agent |
| 路由保护 | Next.js middleware.ts | 服务端检查，不走客户端 |
| 多用户 | 第一阶段就支持 | 每个用户独立数据，邮箱注册即可 |
