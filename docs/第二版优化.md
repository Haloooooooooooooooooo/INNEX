# 第二版优化

## 摘要

本轮优化基于现有代码与现有主骨架推进，不做重建，不另起平行系统。

系统主链路保持为：

`capture -> parse -> internalize -> graph -> QA`

整体目标不是一次性重做，而是按阶段把以下四类资产理顺并增强：

- 原文资产
- 内化资产
- 图谱资产
- QA 资产

其中：

- `capture / parse / internalize / graph build` 继续以 workflow 为主
- `QA` 升级为 agent-guided RAG

## 实施原则

- 只做增量优化，不推翻现有表与主 API
- 先理清数据边界，再增强智能能力
- 每个阶段只解决一个主复杂度
- 长期资产、中间资产、运行时资产分开思考
- 每阶段都应能单独验证，不要求后续阶段全部完成后才见效

## Phase 1：数据与资产边界整理

### 目标

明确现有对象职责，避免上游对象继续承担过多下游责任。

### 核心结论

- `capture_items` 是收录入口对象、原文快照入口对象、流转状态对象、轻解析状态对象
- `notes` 是正式内化资产
- `note_relations` 是正式图谱资产
- `note_chunks` 是中间检索资产
- `capture_items.summary/tags` 是过渡资产
- `notes.summary/tags/concepts` 是正式资产
- `raw_content` 是聚合主视图，不是唯一原文

### 本阶段要落下的内容

- 明确现有字段职责，不急于删字段
- 明确哪些字段仍保留在 `capture_items` 顶层
- 明确哪些内容只是 capture 阶段版本，后续由 note 阶段正式覆盖
- 明确 `raw_content` 与 `note.content` 的层次区别：
  - `raw_content` = 材料层
  - `note.content` = 理解层
- 明确当前默认仍是一条 capture 对应一篇主 note，但不要写死未来永远只能一对一

### 本阶段产出

- 一份清晰的数据对象职责表
- 一份字段分层说明
- 一份长期资产 / 中间资产 / 运行时资产的落位说明

## Phase 2：原文资产与切片基础设施补强

### 目标

让“原文是底座”在系统中真正可用，而不是只停留在概念上。

### 核心结论

- 录入阶段拿到的原始材料要尽量保留
- 原文按来源区分，而不是只保留单个混合字段
- 系统需要两套切片：
  - 内化切片
  - 原文切片

### 本阶段要落下的内容

- 明确原文资产的来源类型：
  - 用户输入文本
  - URL 正文
  - 附件提取文本
  - OCR / 图片补充文本
  - 后续可扩展的视频补充文本
  - 用户理解
- 明确哪些来源内容属于主原文，哪些属于补充原文
- 为原文资产建立可回溯结构，保证后续 QA 和图谱能回到来源
- 补原文切片基础设施，用于：
  - 证据支撑
  - 引用回溯
  - QA 校验
- 保留现有 `note_chunks`，作为知识定位型切片，不与原文切片混用

### 本阶段产出

- 原文来源分类规则
- 原文聚合规则
- 原文切片与内化切片的职责说明

## Phase 3：内化层升级

### 目标

把内化从“生成一篇笔记”升级为“知识升级枢纽”。

### 核心结论

- 内化不仅生成笔记，还负责补解析、正式摘要/标签/概念生成、正式资产产出
- 大文件或 deferred 内容允许在内化阶段补解析
- 内化模型优先级：
  - 忠于原文
  - 有洞察和关联

### 本阶段要落下的内容

- 明确内化输入分层：
  - 主输入：原始正文主体
  - 辅助输入：来源补充内容
  - 引导输入：`my_understanding`
- 固化内化正文结构：
  - 核心内容
  - 关键概念 / 关键信息
  - 原文支持要点
  - 推断与延展
  - 可关联方向
- 明确事实层与推断层显式分开，不能混写
- 正式产出以下资产：
  - 正式正文
  - 正式摘要
  - 正式标签 / 概念
  - 事实层片段
- 推断层片段先至少在正文中分区，后续再决定是否资产化
- 单独为内化讨论模型选型与 prompt 设计，不默认沿用 capture 阶段策略

### 本阶段产出

- 内化输入规范
- 内化输出结构规范
- 内化正式资产清单
- 内化模型 / prompt 调优目标清单

## Phase 4：图谱层增强

### 目标

先把图谱做成“可用的图谱”，而不是“关系很多但不可信的图谱”。

### 核心结论

- 当前阶段图谱主节点继续以 `notes` 为主
- 关系类型先稳定为：
  - `related`
  - `supports`
  - `example_of`
- 暂不引入 `contradicts`

### 本阶段要落下的内容

- 保持图谱节点与正式 note 资产对齐
- 关系生成采用三段式：
  - 粗筛召回
  - 模型判型
  - 证据落库
- 粗筛信号优先使用：
  - embedding similarity
  - tags / concepts overlap
  - title / summary / keyword overlap
- 关系判型优先看：
  - 正式摘要
  - 正式标签 / 概念
  - 事实层片段
- 必要时补：
  - 核心正文
  - 原文片段
- 每条边都具备：
  - `relation_type`
  - `confidence`
  - `evidence_summary`
- 关系发现主要依赖内化资产
- 关系证据尽量回到原文资产

### 本阶段产出

- 图谱关系最小集合规范
- 图谱关系判定输入规范
- 图谱边置信度与证据摘要规范

## Phase 5：QA 检索与 Agent 决策升级

### 目标

把 QA 从普通检索问答升级为 agent-guided RAG。

### 核心结论

- QA 不是基础版单段式 RAG
- QA 以 RAG 为骨架，以 agent 做检索与扩展决策
- 图谱是增强器，不是默认首跳检索入口

### 本阶段要落下的内容

- 固化检索顺序：
  - 先查内化资产
  - 再补原文资产
  - 不够再扩图
- 建立完整 QA 意图识别分类：
  - fact query
  - concept explanation
  - summary / synthesis
  - comparison / distinction
  - action advice
  - evidence strengthening
  - example / case request
  - related-topic expansion
- 建立多轮追问判断：
  - 不把连续消息直接等同于相关追问
  - 用语义连续性判断是否继承上文
- 建立图谱扩检索策略：
  - support-seeking -> `supports`
  - example-seeking -> `example_of`
  - related expansion -> `related`
- 建立停止规则：
  - 证据够就停
  - 收益下降就停
  - 继续扩也补不齐证据时明确保守处理

### 本阶段产出

- QA 检索顺序规范
- QA 意图分类规范
- QA 追问判断规范
- QA 图谱扩检索与停止规则规范

## Phase 6：回答生成与 QA 模型/Prompt 优化

### 目标

让回答既可信、又自然，不像检索片段拼接。

### 核心结论

- QA 按双角色心智设计：
  - 决策角色
  - 回答角色
- 当前阶段允许同一模型承担两种角色
- 回答层最重要的不是“显得聪明”，而是“证据忠实地整合并自然表达”

### 本阶段要落下的内容

- 决策角色优先优化：
  - 稳定分类
  - 保守判断
  - 扩图与停止决策
- 回答角色优先优化：
  - 忠于证据
  - 多源整合
  - 自然解释
  - 边界表达
- 将 QA prompt 分为两类：
  - 决策 prompt
  - 回答 prompt
- 决策 prompt 要求：
  - 短
  - 硬
  - 结构化
  - 非聊天化
- 回答 prompt 要求：
  - 先回答
  - 再说明依据
  - 必要时补关联信息
  - 最后表达边界 / 不确定性
- 回答结构稳定为：
  - 结论
  - 依据
  - 关联补充
  - 边界 / 不确定性
- 回答必须像人说的话，不能像检索片段重排
- 不确定性采用后置边界说明方式表达，不前置弱化整段回答

### 本阶段产出

- QA 决策 prompt 规范
- QA 回答 prompt 规范
- QA 回答风格与边界表达规范
- QA 回答模型能力优先级说明

## 推荐展开顺序

### 第一批先展开

- `Phase 1`
- `Phase 2`

原因：

- 它们决定后面所有层的数据边界与输入基础

### 第二批展开

- `Phase 3`

原因：

- 内化是图谱与 QA 的正式知识入口

### 第三批展开

- `Phase 4`
- `Phase 5`

原因：

- 图谱和 QA 的联动关系在这两个阶段完成

### 最后一批展开

- `Phase 6`

原因：

- 回答生成、模型分工与 prompt 优化适合在前面骨架稳定后收尾

## 默认前提

- 不重建现有主骨架
- 不另起平行系统
- 优先扩展现有对象、流程与 API
- 当前默认仍按“一条 capture 对应一篇主 note”的产品心智推进
- 视频解析不作为当前实施前提，可延后进入后续增强阶段

## 后续写详细计划时的要求

- 继续按 Phase 1-6 顺序展开
- 每阶段只解决一个主复杂度
- 每阶段都给出可验证结果
- 不同时横跨多层做大改
- 细化时优先贴当前代码结构，而不是写成理想化重构方案
