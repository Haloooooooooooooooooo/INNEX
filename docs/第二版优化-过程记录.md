## 优化背景

第二版优化是在现有实现基础上做增量升级，不重建主骨架，继续沿用：

`capture -> parse -> internalize -> graph -> QA`

本轮先从 `Phase 1` 和 `Phase 2` 开始落地，核心目标是：

- 理顺数据与资产边界
- 补强原文资产与切片基础设施
- 让录入、解析、内化的链路在不推翻现有代码的前提下变得更稳

## 8. 实现小红书图文链接按图片数量分流

已修改：

- [parse-url/route.ts](/e:/my_vibecoding/INNEX/apps/web/app/api/parse-url/route.ts)
- [capture-items/route.ts](/e:/my_vibecoding/INNEX/apps/web/app/api/capture-items/route.ts)
- [internalize/route.ts](/e:/my_vibecoding/INNEX/apps/web/app/api/internalize/route.ts)

实现目标：

- 针对“小红书图文笔记文字很少、干货主要在图片中”的场景补一条最小可用链路

本次落地规则：

- 如果识别到小红书链接图片数 `<= 5`
  - 录入阶段直接对全部图片做 OCR
  - 让 summary / tags / raw_content 尽量同时吸收文字和图片内容

- 如果识别到小红书链接图片数 `> 5`
  - 录入阶段不全量 OCR
  - 只保留页面文字、图片数量、deferred 标记和图片 URL 列表
  - 把全量图片 OCR 延后到内化阶段

录入阶段具体做法：

- `parse-url` 新增图片 URL 提取能力
- 返回：
  - `image_urls`
  - `image_count`
  - `platform`
- `capture-items` 根据图片数分流：
  - 小于等于 5 张时，抓远程图片并 OCR
  - 大于 5 张时，写入 deferred 标记

内化阶段具体做法：

- 如果 `parse_debug` 中包含：
  - `url_platform = xiaohongshu`
  - `image_ocr_deferred = true`
  - `deferred_image_urls`
- 则在内化前先把远程图片 OCR 成 `capture_item_sources`
- 再进入原有内化流程

阶段性结果：

- 小红书图文已不再只能“读链接文字”
- 系统开始具备“少图录入即识别、多图延后内化补全”的分流能力

## 过程中遇到的主要问题

## 1. `capture_item_sources` 新能力接入后，Supabase schema cache 未及时识别

实际日志表现：

- `sources-persist-failed`
- `reload-failed`
- `Could not find a relationship between 'capture_items' and 'capture_item_sources' in the schema cache`

问题本质：

- migration 文件已经在本地代码中存在
- 但远端数据库或 PostgREST schema cache 未及时反映新表/关系

这导致：

- `capture_item_sources` 插入不稳定
- `capture_items` reload 时带 `sources:capture_item_sources(*)` 查询报错

应对策略：

- 不让主链路因为新能力未就绪而失败
- 对 `sources` 相关查询和写入做向后兼容降级

处理结果：

- 即使 schema cache 未刷新，录入主流程仍可继续
- `sources` 能力暂时缺失时，系统自动 fallback 到旧行为

---

## 2. 错误日志一度只显示 `[object Object]`

问题表现：

- `sources-persist-failed` 的 error 输出不可读

困难：

- 这样很难快速判断到底是：
  - 表不存在
  - schema cache 问题
  - 关系不可见
  - 还是别的 RLS/SQL 错误

处理方式：

- 增加 Supabase 错误对象格式化逻辑

结果：

- 错误日志变得更可读
- 后续排查 schema cache 问题更容易

---

## 3. 文档类录入的 summary 质量不稳定

问题表现：

- 有时超时
- 有时太长
- 有时直接把 Markdown 结构带进来

困难：

- 这里不是单一问题，而是多个因素叠加：
  - prompt 乱码
  - 超时太短
  - 缺少后处理
  - fallback 太原始

处理方式：

- 修 prompt
- 放宽 summary retry/timeout
- 增加摘要清洗
- 增加 heuristic fallback

结果：

- 录入阶段摘要已明显收紧
- 但后续还需要继续观察实际样本质量

## 本轮采用的总体处理策略

这次优化没有选择“推倒重来”，而是遵守了以下策略：

1. 只做增量增强
- 在现有对象和 API 上扩展
- 不另起平行系统

2. 优先保证主链路稳定
- 新能力未完全就绪时，允许 graceful fallback
- 不因为 `sources` 基础设施未准备好就让录入失败

3. 把错误分层处理
- 模型慢：放宽 timeout + fallback
- prompt 脏：修 prompt
- schema cache 未刷新：降级兼容
- 输出太长：后处理清洗

4. 每改一轮都做构建验证
- 多次执行 `apps/web` 下的 `npm.cmd run build`
- 当前相关改动均已通过构建

## 2026-05-29 补充复盘：小红书图文链路（Phase 2）收敛结果

### 本轮进度
- 已打通「小红书链接 -> 图片抓取 -> OCR -> 入库 sources -> 聚合到 raw_content」链路。
- 最终验收样本显示：
  - `url_platform = xiaohongshu`
  - `url_image_count = 4`
  - `inline_image_ocr_count = 4`
  - `notes` 包含 `xhs_image_ocr_succeeded:4`、`xhs_external_crawler_used:4`
- 说明：本地外部爬虫回退链路生效，且已能稳定拿到图文图片并完成 OCR。

### 遇到的主要困难
- 1) 小红书反爬与登录墙：
  - 直接页面抓取时经常拿到登录页文案，或拿不到真实图片 URL。
- 2) 候选图片 URL 噪音高：
  - 出现大量 `application/javascript` / `text/css` 伪图片链接。
- 3) OCR 模型兼容与稳定性：
  - 部分模型对图片请求返回 400，导致批量 OCR 失败。
- 4) OCR 内容重复严重：
  - 识别结果中出现大量循环行，导致 `raw_content` 膨胀、噪音高。

### 解决方案（已落地）
- 1) 三段式抓取回退：
  - 静态提取 -> 渲染提取 -> 外部 crawler 接口（本地可用，后续可替换部署版）。
- 2) 平台识别与登录墙处理：
  - 对 xiaohongshu 强制平台识别；
  - 对登录墙 OCR 结果识别并丢弃（避免脏数据入库）。
- 3) 远程图片抓取增强：
  - 强化请求头、403 重试、content-type 校验、大小限制。
- 4) OCR 结果清洗去重：
  - 增加轻量 normalize：
    - 去连续重复行；
    - 同一行全局限次；
    - 压缩短周期循环段；
  - 多图 OCR 块去重后再拼接到 `raw_content`。
- 5) 内部接口鉴权修复：
  - `capture-items` 调用 `parse-url` 时透传 cookie，修复 401。

### 可复用经验（建议沉淀为团队规范）
- 1) 反爬平台接入必须设计「多级回退」，不要押注单一抓取方式。
- 2) URL 候选必须先做内容类型过滤，再进 OCR；否则成本高且噪音大。
- 3) OCR 不是“识别完就用”，必须配套清洗与去重层。
- 4) 新能力上线时优先「可降级」：
  - schema/cache/外部依赖异常时，主链路仍要可用。
- 5) 验收标准要数据化：
  - 以 `parse_debug` 的关键字段作为通过条件（平台、图片数、OCR 成功数、回退链路标记）。

### 当前状态判断
- Phase 2（小红书图文原文资产采集）已达到“可用且可验收”的状态。
- 仍建议在 benchmark 样本池上继续跑 1~2 轮稳定性回归（不同链接、不同图数、不同网络条件）。
- 可进入 Phase 3（内化层升级），同时保留本链路的灰度观察与日志监控。

## 2026-05-29 补充复盘：Phase 3（内化层升级）收敛结果

### 本轮进度
- 已完成“无原文不内化”的硬约束：
  - 内化阶段若无法拿到 primary source（`raw_content + 附件抽取文本`），直接 422 阻断。
- 已完成内化阶段附件抽取增强：
  - parser-service 解析失败时，增加 URL 拉取与 Supabase Storage 下载双通道兜底。
- 已完成 dry-run 行为修正：
  - `dry_run=true` 也执行真实原文抽取，不再出现“诊断为空”。
- 已完成内化状态会话化：
  - 启动内化后关闭抽屉不取消流程；重开同记录自动回到内化页。
- 已完成内化结果回写增强：
  - 保存后即时同步 `status/summary/tags` 与 AI 笔记内容，减少“保存后无变化”错觉。

### 遇到的主要困难
- 1) 旧兜底路径会“伪成功”：
  - 无原文时仍可生成通用笔记，影响数据可信度。
- 2) 状态一致性复杂：
  - 列表、详情、草稿、后台异步提交存在时序竞争。
- 3) 附件可读路径不稳定：
  - `storage_path` 可能是公网 URL、对象路径或临时不可访问路径。
- 4) 标签生成条件误判：
  - `['-']` 被视为已有标签，导致内化阶段不触发标签生成。

### 解决方案（已落地）
- 1) 强制原文门禁：
  - 新增最小原文长度阈值，未达到则返回 `INTERNALIZE_NO_CONTENT`。
- 2) 抽取诊断可观测化：
  - `blocked_no_primary_source` 日志新增 `attachment_diag`，细化到 parser、URL fetch、storage download、extract 各步骤。
- 3) dry-run 与正式内化一致化：
  - dry-run 不再跳过附件解析，预览结果与正式执行口径一致。
- 4) UI 会话化收敛：
  - 内化会话期间列表按钮持续显示“内化中”；关闭后重开自动回到内化页；仅“返回详情/保存完成或失败”结束会话。
- 5) 标签/摘要补全策略修正：
  - 仅在空值场景生成；并把 `- / — / 暂无标签` 归为无效标签；模型失败时启发式兜底补全。

### 可复用经验（建议沉淀为团队规范）
- 1) “可用”不等于“可信”：涉及知识资产入库，必须先保证原文可追溯。
- 2) 内化类功能应采用“会话化状态机”而非单点按钮状态。
- 3) 解析链路必须有分步骤诊断字段，否则无法高效定位根因。
- 4) 占位值（如 `-`）必须纳入业务语义校验，否则会误伤后续流程。
- 5) dry-run 与正式执行必须同口径，避免预览与落库行为分叉。

### 验收口径（本轮建议）
- 1) `no_source_but_success_rate = 0`
- 2) `internalize_parse_success_rate >= 98%`
- 3) 内化会话恢复成功率（关闭后重开）= 100%
- 4) 有效标签覆盖率（排除占位值）>= 98%
- 5) 关键链路日志具备可定位字段（`parse_status/parse_error_code/attachment_diag`）

### 当前状态判断
- Phase 3 已进入“能力可用 + 规则收敛”的状态。
- 仍建议在 benchmark 样本池上继续做 1~2 轮稳定性回归（重点覆盖：URL 无正文、私有附件、大 PDF、低质量 OCR）。
- 在 QA 全量优化前，可先冻结 Phase 3 规则，进入“稳定性验证与 badcase 回灌”阶段。

### 本轮采用的图谱关系规范（Phase 4 验收输出）
1. 关系类型规范
- 当前仅允许三类关系：`related`、`supports`、`example_of`。
- 不引入 `contradicts`，避免在证据体系与冲突判定策略尚未完备前放大误判风险。

2. 关系生成规则（V1）
- 三段式固定流程：
  - 粗筛召回：embedding 相似度召回候选 note。
  - 判型：结合 tags/concepts overlap 做轻判型（当前自动产边主要为 `related / supports`）。
  - 证据落库：将 confidence 与 evidence_summary 一并入库。
- 主输入优先：note 摘要、标签/概念、事实片段。
- 辅助输入：正文与原文片段（由上游内化资产组织提供）。
- `my_understanding` 与推断层内容不作为主判定依据。

3. 边证据与置信度规则（V1）
- 每条自动关系边至少包含：
  - `relation_type`
  - `confidence`
  - `evidence.evidence_summary`
- 置信度采用分层解释，不追求过度精细数值：
  - 高：`>= 0.75`
  - 中：`0.60 ~ 0.75`
  - 低：`< 0.60`
- evidence_summary 优先引用：共享概念、事实片段线索、原文支持片段索引、简短判定理由。

4. 与 QA 的结构价值确认

## 2026-05-29 追加修复：图谱出现“有节点无边”

  - 当 embedding 召回未产边时，改用结构化重叠信号补边：
    - tags/concepts overlap
    - title/summary keyword overlap
  - 兜底产边仍写入 Phase4 证据字段：
    - `relation_type`
    - `confidence`
    - `evidence.evidence_summary`
  - 关系类型仍限定在当前稳定集合：`related/supports/example_of`（兜底当前产 `related/supports`）。

### 结果
- 即使 embedding 链路暂时不可用，也能避免“图谱完全断开”。
- 构建通过（`npm run build`）。

### 后续建议
- 继续观察：若 embedding 恢复，应优先使用向量召回结果；兜底仅作为保底路径。
- 在 benchmark 结果表中，单独记录是否触发 `structured_overlap_fallback`，便于统计真实向量链路稳定性。

## 图谱优化可复用经验（阶段复盘沉淀）

以下内容用于把本轮图谱优化沉淀成可复用方法，后续新样本、新模型或新业务主题可直接套用。

### 1. 核心方法论（先召回，再控误连）

1. 先做高召回候选，再做强判型重排，不要一开始就把阈值卡得太死。
2. 建边是两阶段任务：
   - 阶段A：语义候选召回（向量 + 结构信号）
   - 阶段B：关系判型与分层入库（LLM + 分类型阈值）
3. 关系质量要分层管理，不能只做“连/不连”二值判断。

结论：
- 召回低时，不要先暴力降阈值；优先扩候选与增强语义特征。
- 误连高时，不要先缩候选；优先分关系类型阈值与证据约束。

### 2. 建边主链路（当前稳定版本）

当前主链路：
1. 新 note 内化后生成 `note_chunks`。
2. 关系召回优先使用：`事实片段 -> 核心片段 -> 关联方向片段`。
3. `推断/延展片段`在关系主召回中降权（避免推断污染建边）。
4. 候选进入 `v4-pro` 关系判型器（固定 JSON 输出）。
5. 按 `relation_type` 分阈值入库（supports/example_of/related）。
6. 低证据关系降级为 `weak_related/fallback`，而不是直接强连。

### 3. 这轮最关键的有效改动

1. 从“单阈值”改为“分关系类型阈值”。
2. 引入 `weak_related/fallback` 弱边分层（可追踪、可回滚、可视化可解释）。
3. 加入同义词归一（Prompt/提示词、AB测试、Agent、RAG），显著改善语义近义召回。
4. 将“可关联方向”纳入召回信号，提升跨表述连接能力。
5. 图谱展示层区分线型（强边实线、弱边虚线、fallback淡虚线），提升可读性与调参效率。

### 4. 踩坑经验

1. 只靠词面重叠很容易误连（尤其是泛词：AI、产品经理、系统、方法）。
2. 只靠严格阈值很容易漏连（语义相关但表达不同）。
3. 把推断段和事实段等权使用，会导致“看似合理但证据薄弱”的关系增多。
4. 如果不区分 LLM 判型与 fallback 来源，后续很难定位问题根因。

### 5. 推荐评测口径（固定）

每轮至少评估：
1. `edge_count` 与 `relationTypeCounts`
2. 强边准确率（supports/example_of）
3. 弱边可用率（weak_related/fallback）
4. 漏连与误连 badcase 数量
5. 证据摘要可用性（evidence_summary 是否可解释）

实践建议：
- 每轮都写入 `benchmark-结果记录表`，并同步 badcase。
- A/B 对比必须用同一批样本，避免“换数据导致误判优化效果”。

### 6. 样本口径经验

1. 关系评测按“内容语义”而不是“平台来源”定义。
2. 同平台样本也可能高相关（例如 Agent 主题组）。
3. 独立样本要明确标注（如 S11），避免把“本不该连”的样本误算为漏连。

### 7. 默认策略建议（后续复用）

1. 主模型：`v4-pro`（先稳定后对照）。
2. 模式：先“保守模式”，再逐步放开。
3. 候选：先提高候选覆盖，再做分层过滤。
4. 入库：允许弱边入库，但默认弱展示、低权消费。
5. QA：优先消费强边，弱边仅在证据不足时补充扩展。

### 8. 后续可复用扩展点

1. 增加 `orphan detection`（孤岛节点识别）并进行弱边补全。
2. 增加多跳扩展（用于 QA 检索阶段，不直接污染主图）。
3. 引入模型对照组（如 OpenAI）做能力/成本/时延三维横评。
4. 建立“关系判型回放集”，对 badcase 做持续回归。

---

复用原则总结：
- 图谱优化不是一次调阈值，而是“召回-判型-分层-评测-回放”的闭环工程。
- 只要这套闭环保持，换新样本或换模型时都能稳定迭代。

## 2026-05-31｜根因：embedding 主路径不按判型阈值拦边（过度连接真因）+ 节点大小改 degree

用户反馈 R10 后“改阈值没用、仍过度连接、全橙、节点大小一样”。导出 `innex-graph-1780205762670.json`（19 节点 / 148 边，related=135/example_of=10/supports=3，confidence 大量在 0.55~0.65）。

1. confidence 分布里存在 0.55、0.60 的 related 边 → 若 related 阈值 0.63 生效，这些边不该入库 → 阈值没生效。
2. 用户确认：改 .env.local 后**没重启 dev server**（Next.js 仅启动时读 env）→ 这次重内化用的还是旧 0.58。
3. 更深层 bug：`build_relations` 的 embedding 召回主路径是“召回到就建边”——LLM 判型仅用来“升级”类型/置信度，**从不拒绝建边**；唯一的 `continue` 兜底还被 `RELATION_CONSERVATIVE_MODE` 限定（balanced 下永不触发）。所以 `RELATION_TYPE_THRESHOLDS.related` 对这条主路径根本不起拦截作用，入库实际门槛只是召回阈值 `RELATION_MATCH_THRESHOLD=0.56`，bge-m3 下几乎全过 → 必然过度连接、且边多保留 lite 默认 `related`。

修复（`apps/web/app/api/internalize/route.ts`，纯逻辑，不改 prompt/模型）：
1. embedding 主路径赋予 LLM 判型“否决权”：
   - `relation_type === "none"` → `continue`（不建边）。
   - `confidence < max(RELATION_LLM_MIN_CONFIDENCE, 该类型阈值)` → `continue`（不建边）。
   - 否则用 LLM 的 type/confidence 建边。
2. 无 LLM 判型（预算耗尽/调用失败）时的 lite 兜底：新增 `RELATION_LITE_SIMILARITY_FLOOR`（默认 0.72），相似度不到该下限不建边，避免无脑全连。
3. 效果：related 阈值（0.63）+ weak 被 `RELATION_LLM_MIN_CONFIDENCE=0.56` 拦下，真正生效；预期边数大幅下降、无关点断开。

颜色/节点大小（回应用户三问，前一轮已部分落地）：

- 节点大小本轮改为按 `degree`（连接数，相对最忙节点归一 14..40）渲染（`nodeDegreeMap`/`maxNodeDegree`），区分度远大于此前“平均 confidence”，且随过滤实时变化；刷新即生效。

### 问题与根因

#### Q5 续问断链（R7-C02 未完全修复）

**现象**：Q4 问"PRD模板包含哪些模块"→正确命中 PRD 笔记。Q5 问"继续上一个PRD话题，给我一个可直接套用的最小模板"→召回了"Agent设计"笔记，完全丢失 PRD 上下文，回答"无法提供"。

**根因**：
1. `FOLLOWUP_REFERENCE_MARKERS` 包含"继续/上一个"，应触发 Stage 3.7 续问复用。
2. 但续问复用（Stage 3.7）只对上轮引用 note 做 `match_note_chunks_in_notes` 补召回——如果主检索（Stage 1/2）已经召回了足够数量的 chunks（来自错误的笔记），续问复用的结果可能被淹没或排序靠后。
3. 核心矛盾：**主检索走 embedding 召回，"最小模板"这个表述被 bge-m3 匹配到 Agent 设计笔记（也讲方法论/模板），相似度可能高于 PRD 笔记中对应 chunk**。续问复用虽然注入了正确 note 的 chunks，但在最终 fuse 排序中被压下去了。

**修复方向**（检索层，Phase 6 前建议先修）：
- 方案 A：续问场景下，Stage 3.7 复用的 chunks 给予排序加权（类似 `reorderChunksBySessionTopic` 的逻辑，但针对续问复用结果）。
- 方案 B：续问场景下，限制主检索只在上轮命中 note 范围内召回（更激进，可能误伤跨话题续问）。
- 建议先试方案 A，风险更低。

#### Q7 低质片段浪费 token

Q7 检索到 7 条完全不相关片段（面试技巧、Vibe Coding 等），证据分仅 54%。回答层正确判定"暂无直接关联"，但这些片段白白占用了回答模型的 context。

**优化方向**：证据分极低（<55%）且无强命中时，可提前截断不送入回答模型，节省 token 并降低噪音干扰风险。

## 2026-05-31｜Phase 6 修复：QA 回答 markdown 渲染

### 问题

用户反馈 AI 回答只显示原始 markdown 源码（`**粗体**`、`### 标题`、`- 列表`、`` `代码` `` 等），没有渲染成富文本，阅读体验差。

### 根因

`answer-display.tsx` 用 `whitespace-pre-wrap` 把 answer 当纯文本展示，没有任何 markdown 解析。且 `prose` 类从未生效——`@tailwindcss/typography` 插件没有安装配置。

### 已落地

1. 安装 `react-markdown`（10.1.0）+ `remark-gfm`（GFM 扩展：表格/删除线/任务列表）。
2. 安装 `@tailwindcss/typography`（0.5.19），在 `globals.css` 中以 Tailwind v4 语法 `@plugin "@tailwindcss/typography"` 引入。
3. 改造 `answer-display.tsx`：
   - `whitespace-pre-wrap` 纯文本 → `<ReactMarkdown remarkPlugins={[remarkGfm]}>` 渲染 markdown。
   - 自定义 prose 样式适配项目色系：链接橙色、code 底色 paper 前景 accent、标题 ink。
   - 引用依据卡片与保存按钮逻辑不变。

### 效果

- 回答中的 markdown 语法（标题、加粗、列表、链接、行内代码、代码块等）正常渲染为富文本。
- 用户不再看到 `**`、`###` 等 markdown 源码标记。

### 涉及文件

- `apps/web/components/qa/answer-display.tsx`
- `apps/web/app/globals.css`
- `apps/web/package.json`

### 验证

- `npm run build` 通过（Compiled successfully，TypeScript 无 error）。
