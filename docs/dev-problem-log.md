## 2026-05-22｜录入解析模型 TLS 握手失败导致摘要/标签异常

- 模块：收录箱录入 / 解析
- 现象：录入 md 文档时出现 `summary_error` / `tags_error`，并暴露底层 TLS 握手错误。
- 影响：P0（影响用户对“录入是否成功”的判断，严重影响体验）
- 根因：模型网关链路偶发 TLS 握手失败；前端直接展示底层异常；解析链路缺乏主备与补偿。
- 解决方案：
  1. 错误归一化：将底层 SSL/TLS 细节归类为业务级错误码（如 `model_network_error`）。
  2. 前端提示降级：录入成功时只提示“摘要/标签暂不可用，已先完成收录”。
  3. Parse 主备模型切换：同一请求内自动尝试备用 provider。
  4. 失败补偿重试：录入成功后异步触发 `retry-parse` 并回写 summary/tags。
- 思路复盘：
  - 先保证“录入成功率”，再追求“解析完整率”。
  - 把“同步强依赖”改为“同步成功 + 异步补偿”可以显著提升稳定性。
- 后续动作：引入队列化重试（指数退避 + 最大重试次数 + 可观测面板）。
- 关联文件：
  - `apps/web/lib/parse/generator.ts`
  - `apps/web/components/inbox/quick-capture.tsx`
  - `apps/web/app/api/capture-items/route.ts`
  - `apps/web/app/api/capture-items/[id]/retry-parse/route.ts`

## 2026-05-22｜录入阶段“解析成功但最终500”与模型不稳定的联合问题（已解决）

- 模块：收录箱录入 / 解析 / 保存
- 现象：
  - 文档/链接录入时，`parse-result` 显示 `model_summary_succeeded=true`、`model_tags_succeeded=true`，但 `POST /api/capture-items` 仍返回 500。
  - 图片/部分文档场景中，模型偶发 `empty response` 或标签 JSON 格式不完整导致 tags 失败。
  - 用户主观感受为“刚刚还能生成，下一条又不行”。
- 影响：P0（主链路不稳定，用户对系统可用性失去信心）
- 根因：
  1. 模型侧：DeepSeek 在高并发/瞬时波动下会出现空返回或非严格 JSON 输出（尤其 tags）。
  2. 代码侧：保存阶段对“附件写库失败/回查失败”是硬失败，导致即使解析完成也整条 500。
  3. OCR侧：多模态返回 `content` 可能是数组结构，旧逻辑只按字符串读取，导致误判“无文本”。
- 解决方案：
  1. DeepSeek 文本调用改为官方 `/chat/completions` 直连，减少兼容层不确定性。
  2. 模型稳定性增强：
     - 对 `empty_response/429/5xx/timeout` 增强重试与退避。
     - 同通道模型回退（`deepseek-v4-flash -> deepseek-v4-pro`）。
  3. 标签解析容错：
     - 先严格 JSON 解析；
     - 失败时采用宽松分割解析（逗号/换行/中文顿号），避免因格式细节丢标签。
  4. OCR兼容：
     - 同时支持字符串与数组式 `message.content` 读取；
     - 图片无 OCR 文本时启用多模态“看图生成摘要+标签”兜底。
  5. 保存链路稳态化：
     - 附件元数据写入失败不再中断整条收录；
     - 最终回查失败返回已创建记录并附带 `warnings`，不再 500。
  6. 可观测性：
     - 增加/强化 `parse.model.failed`、`capture-items.parse-result`、`retry-parse.*` 结构化日志，便于快速判因。
- 思路复盘：
  - “解析成功但保存失败”本质是链路耦合问题，必须将保存阶段从硬失败改为可降级。
  - 模型结果格式不能假设绝对规范，生产环境必须内置容错解析。
  - 先保证“记录成功落库”，再在异步链路补齐增强能力，是更稳的产品策略。
- 后续动作：
  1. 将 `parse_debug` 列迁移状态统一（避免 `compat-no-parse-debug` 长期存在）。
  2. 增加后台“模型可用性监控面板”（按阶段统计 empty/timeout/format-error）。
  3. 评估将异步重试改为队列任务（可观测、可回放、可限流）。
- 关联文件：
  - `apps/web/lib/llm/client.ts`
  - `apps/web/lib/parse/generator.ts`
  - `apps/web/app/api/capture-items/route.ts`
  - `apps/web/app/api/capture-items/[id]/retry-parse/route.ts`

## 2026-05-24｜PDF 录入解析长链路不稳定（从不可用到可用的完整排障）

- 模块：收录箱录入 / PDF 解析 / 模型摘要标签 / 本地开发环境
- 现象：
  - 我上传图片时，记录能创建，但 `model_summary_succeeded/model_tags_succeeded` 与真实状态不一致（显示成功但实际是兜底文案）。
  - 我上传 PDF 时频繁出现 422（`PDF_PARSE_REQUIRED_FAILED`），随后又出现“录入中长时间不返回”。
  - Next.js dev（Turbopack）还出现缓存损坏，导致本地服务不稳定。
- 影响：P0（核心录入链路不稳定，直接影响我继续做 vibe coding 的节奏）

- 我遇到的核心困难点：
  1. 故障不是单点，而是“多段链路同时波动”：前端 dev 缓存、PDF 服务、OCR worker、模型超时交织在一起。
  2. 日志里“成功”和“失败”信号混在一起，容易误判（例如兜底文案把状态伪装成成功）。
  3. parser-service 首次运行要下载模型，初始化慢，导致我主观上觉得“卡死”。
  4. 同一个问题在不同阶段表现不同：先 422、再 201 但内容空、再 201 且有内容。

- 我是怎么一步步解决的（我的动作）：
  1. 先盯结构化日志，不猜：
  - 我持续看 `[capture-items.parse-result]` 的 `input_source/extracted_chars/notes/model_*`。
  - 我用 `notes` 逐条定位：`pdf_parser_service_unavailable`、`pdf_ocr_failed`、`required_pdf_failed`、`image_*_forced_fallback`。
  2. 先保主链路“可收录”：
  - 我把“基础设施失败”从硬拦截里剥离，确保先 201 落库，不因 OCR/服务故障直接 422。
  3. 再修状态真实性：
  - 我把图片兜底逻辑改成“可兜底但不伪造成功状态”，让 debug 能反映真实模型结果。
  4. 解决可重试能力：
  - 我让 `retry-parse` 支持 `raw_content=null` 的图片，从附件 URL 重跑视觉识别。
  5. 解决 parser-service 不稳定：
  - 我重建 venv、单独启动 parser-service（切到 8012）、并把 web 端 `PARSER_SERVICE_URL` 对齐到 8012。
  - 我把 parser-service 改成延迟初始化，避免启动阶段卡死健康检查。
  6. 解决“录入中卡住”：
  - 我给 `parsePdfWithService` 加了可配置超时 `PARSER_SERVICE_TIMEOUT_MS`，避免无限等待。
  7. 解决本地环境噪音：
  - 我清理了 `apps/web/.next/dev` 与 `apps/web/.next/cache`，修复 Turbopack 缓存损坏。
  8. 最后做质量优化：
  - 我做了文本控制字符清洗与标签泛词过滤，减少 PDF 提取脏字符对摘要/标签质量的影响。

- 我采取的方案（最终落地）：
  1. 可用性优先：硬失败改软降级，先保证记录创建成功。
  2. 真实性优先：debug 状态必须反映“模型真实成功/失败”，不能被兜底掩盖。
  3. 可恢复优先：失败后可重试（图片支持从附件重跑）。
  4. 可观测优先：以 `notes` 为诊断主线，所有关键分支都打可读标记。
  5. 可调优优先：超时、端口、模型都改为可配置，避免每次改代码。

- 经验教训（这次最值钱的）：
  1. 先把“能不能收录”与“能不能高质量解析”拆开处理，先救主流程，再做质量。
  2. 多模块故障时，必须靠结构化日志收敛，不要凭体感判断。
  3. 本地 AI 依赖首次下载是常态，首次慢不等于挂；要区分“慢”与“不可达”。
  4. dev 缓存损坏会伪装成业务 bug，环境层要优先排除。
  5. 兜底策略必须配套真实状态字段，否则后续排障会被误导。

- 下次可复用的操作清单（我可以直接照抄）：
  1. 先看 `capture-items.parse-result`：重点 `input_source/extracted_chars/model_*` 与 `notes`。
  2. 检查 parser-service：
  - `GET /health` 是否秒回；
  - `PARSER_SERVICE_URL` 端口是否对齐；
  - 首次启动是否在下载模型。
  3. 若前端长时间“录入中”：
  - 检查服务端是否有超时保护；
  - 检查是否在等待外部解析服务。
  4. 若本地 Next 异常：
  - 清理 `.next/dev` + `.next/cache` 后重启 dev。
  5. 若模型阶段波动：
  - 提升 tags 重试/超时配置；
  - 保留启发式标签兜底，避免空标签。
  6. 每次修复后都看一条完整新样本，确认从“日志正确”到“业务结果正确”都成立。

## 2026-05-23｜知识图谱首版加载慢、点击后空白、重叠严重（已解决）

- 模块：知识库图谱（`/kb`）
- 现象：
  - 首次进入图谱等待明显，体验“空白等待”。
  - 点击节点后出现图谱接近空白。
  - 节点与标签重叠严重，观感混乱。
- 影响：P0（主功能可用性和可读性受损）
- 根因：
  1. 点击节点时触发了不必要的全量重拉（`activeNoteId` 误入拉取依赖）。
  2. 过滤逻辑把节点集裁剪得过窄，导致“只剩局部甚至看起来空白”。
  3. 力导布局参数偏保守，标签策略不友好，重叠概率高。
- 解决方案：
  1. 去除节点点击导致的图谱重拉，图谱数据与“选中态”解耦。
  2. 调整“仅显示相关关系”为视觉降噪，不再裁剪全图节点集合。
  3. 增加本地图谱缓存（先秒开缓存，再后台刷新最新数据）。
  4. 布局参数优化（更强斥力、更大连线距离、碰撞强度）+ 标签截断与字号调整。
- 思路复盘：
  - 图可视化里“数据刷新”和“视图交互”必须拆开，否则会造成闪烁、空白与性能抖动。
  - 默认尽量保留全图，降噪优先走状态（高亮/置灰），不要直接删数据。
- 后续动作：
  1. 增加“节点数量阈值分层渲染”（大图先主题后展开）。
  2. 增加标签碰撞回避策略（可按缩放级别动态显示密度）。
- 关联文件：
  - `apps/web/app/kb/page.tsx`

## 2026-05-23｜图谱存在重复边与噪声边（已解决）

- 模块：图谱数据接口（`/api/graph`）
- 现象：视觉上出现“重复关系线”或关系密度异常。
- 影响：P1（可读性下降，用户误判关系强度）
- 根因：后端返回的关系中可能包含同一对节点同类型的重复记录，且未过滤自环。
- 解决方案：
  1. 在 `/api/graph` 对边做去重（无向归一键：`a|b|type`）。
  2. 过滤自环边（`source == target`）。
- 思路复盘：
  - 图谱渲染层不应承担数据去重职责，接口层应输出“可直接渲染”的干净数据。
- 后续动作：对入库阶段增加唯一性约束或幂等写入策略，减少后置去重压力。
- 关联文件：
  - `apps/web/app/api/graph/route.ts`

- 根因：提问入口在不同组件各自实现，缺少统一触发约定。

## 2026-05-23｜抽屉问答缺少用户消息回显，导致“像表单而非对话”（已解决）

- 模块：AI问答（右侧抽屉）
- 现象：发送后只显示 AI 回复，不显示用户刚发送内容；用户难以确认“我问了什么”。
- 影响：P1（对话可读性弱，追问效率低）
- 根因：抽屉问答只维护 `qaAnswer` 单值，没有消息数组模型。
- 解决方案：
  1. 新增 `qaMessages` 状态（`user/assistant/at`）。
  2. 发送时先插入用户消息（乐观渲染），再请求 `/api/qa`。
  3. 成功后追加 assistant 消息；失败保留用户消息并显示错误。
  4. 进入提问模式时清空 `qaMessages`，保证单条记录上下文清晰。
- 思路复盘：
  - 聊天 UI 的最小单元是消息流，不是“输入 + 最后答案”。
  - 乐观渲染能显著提升主观响应速度。
- 后续动作：将 `qaMessages` 与会话持久化打通，支持历史恢复。
- 关联文件：
  - `apps/web/components/inbox/inbox-drawer.tsx`
