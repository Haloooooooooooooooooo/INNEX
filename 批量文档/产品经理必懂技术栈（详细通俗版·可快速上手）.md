# 产品经理必懂技术栈（详细通俗版·可快速上手）

核心目标：**不写代码、能沟通、能评估、能判断**，贴合产品经理日常工作（需求评审、和研发对接、方案设计），所有知识点均为“PM高频用到”，避免冗余技术细节，讲不清的补充极简学习链接（无需深入，看10分钟就能懂）。

# A\. 技术栈基础认知（10分钟吃透）

## 1\. 什么是技术栈

通俗理解：**一整套“搭建产品、让产品跑起来”的技术工具组合**，类比“盖房子”——不是只靠砖头（编程语言），还要有水泥（框架）、钢筋（数据库）、装修（前端）、水电（部署），缺一不可。

核心构成（必记）：前端（用户看到的）\+ 后端（背后逻辑）\+ 数据库（存数据）\+ AI（智能能力，可选）\+ 部署/运维（上线后保障）。

核心目标：把产品经理的“想法”（需求），变成用户能实际使用的“产品”（网页/APP），并保障产品稳定运行。

## 2\. 为什么产品经理要懂技术栈

不用记复杂理论，记住5个实际价值（PM日常高频场景）：

- 和工程师高效沟通：不用“鸡同鸭讲”，比如不说“我要一个能自动刷新的页面”，而是说“这个页面需要通过接口实时拉取数据，前端做定时刷新”，研发能快速get需求。

- 评估需求可行性：知道“这个需求能不能做”“做起来难不难”，比如“让用户上传100M视频并实时转码”，能判断出“后端压力大，不能实时，需要异步处理”，避免提无效需求。

- 判断开发成本和周期：比如“简单改个按钮样式”（前端1小时），和“做一个用户登录\+权限管理”（后端\+前端1\-2天），能大概估算工期，避免乱承诺排期。

- 定位问题所在：产品上线后报错，能判断是“前端页面问题”（比如按钮点不动）、“后端接口问题”（比如数据加载失败），还是“数据库问题”（比如数据丢了），快速协调对应研发解决。

- 设计更合理的产品方案：比如知道“缓存能加快访问速度”，就会设计“热门内容优先缓存”的方案；知道“大模型有上下文限制”，就会设计“分段提问”的交互，提升用户体验。

## 3\. 产品经理需要掌握到什么程度（核心红线）

绝对不用会：写复杂代码、调试程序、搭建服务器。

必须会：

- 理解每个技术的“基本原理”（比如“数据库是存数据的”“缓存是加快速度的”）；

- 能看懂研发画的“架构图”（知道哪个模块对应哪个功能）；

- 能理解“数据流”（比如用户点击按钮后，数据从前端传到后端，再到数据库，最后返回前端的整个过程）；

- 知道不同技术方案的“优缺点”（比如“用Redis缓存快，但不能存大量数据；用MySQL能存大量数据，但访问慢”），能和研发讨论取舍。

# B\. Web / 互联网基础（PM最常用，20分钟掌握）

## 1\. 浏览器和服务器（最基础，必懂）

通俗类比：浏览器 = 你手机里的“微信”（入口），服务器 = 微信后台的“机房”（真正处理消息、存数据的地方）。

核心逻辑：**客户端\-服务端模式**——用户通过“浏览器”（客户端）发起请求（比如打开一个网页、点击一个按钮），请求发送到“服务器”，服务器处理后，把结果返回给浏览器，浏览器再展示给用户。

PM重点：知道“用户操作 → 浏览器发请求 → 服务器处理 → 浏览器展示”这个流程，就能理解“为什么有些操作会有延迟”（服务器离用户远、处理慢）。

## 2\. 域名、DNS、IP（不用记技术细节，记作用）

- 域名：就是你平时输的网址（比如www\.baidu\.com），作用是“好记”——相当于“小区名字”，不用记复杂的门牌号。

- IP：服务器在网络中的“唯一地址”，相当于“小区门牌号”（比如192\.168\.1\.1），浏览器最终要通过IP找到服务器。

- DNS：相当于“导航仪”，作用是“把域名翻译成IP”——你输www\.baidu\.com，DNS就会告诉浏览器“这个域名对应的IP是xxx\.xxx\.xxx\.xxx”，浏览器才能找到百度的服务器。

PM重点：知道“域名要备案”“DNS解析可能有延迟”（比如刚改完域名，有些用户要等10分钟才能打开），避免上线时踩坑。

## 3\. HTTP / HTTPS（请求协议，PM必懂）

通俗理解：HTTP/HTTPS是“浏览器和服务器之间的沟通规则”，相当于“两个人说话要讲同一种语言”，否则无法沟通。

核心区别：HTTPS比HTTP多了“加密”，更安全——比如用户输入密码、支付信息，必须用HTTPS，否则信息可能被窃取；普通的文章展示，用HTTP也可以（现在大多都用HTTPS）。

常用请求方式（必记，PM写需求时会用到）：

- GET：获取数据（比如打开网页、查看用户信息）——特点：请求参数会显示在网址上，不能传敏感数据（比如密码）。

- POST：提交数据（比如登录、注册、提交表单）——特点：请求参数不显示在网址上，安全，适合传敏感数据。

- PUT/PATCH：修改数据（比如修改用户名、更新订单状态）——PUT是“全量修改”，PATCH是“部分修改”（PM不用区分太细，知道是“修改数据”即可）。

- DELETE：删除数据（比如删除订单、删除好友）。

学习补充（10分钟看懂）：[MDN HTTP 基础（只看“HTTP 是什么”“请求方法”部分）](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Overview)

## 4\. Request / Response（请求和响应，PM看接口文档必懂）

通俗理解：Request = 浏览器发给服务器的“请求包”（里面有“我要什么”“我是谁”）；Response = 服务器返回给浏览器的“响应包”（里面有“你要的数据”“操作结果”）。

核心组成（必记，看接口文档时能对应上）：

- 请求头：相当于“请求的附加信息”，比如“我是Chrome浏览器”“我已经登录了（携带Token）”“我要的数据格式是JSON”。

- 请求体：相当于“请求的具体内容”，比如POST请求时，里面是“用户名=xxx\&amp;密码=xxx”。

- 状态码（最常用，必记4个）：
        

    - 200：成功（操作没问题，服务器正常返回数据）；

    - 401：未授权（用户没登录，或者登录过期，需要重新登录）；

    - 404：找不到（请求的地址错了，比如网址输错、接口路径错了）；

    - 500：服务端错误（服务器出问题了，不是用户操作的问题，需要研发排查）。

- 响应数据格式：通常是JSON（一种key\-value格式，比如\{\&\#34;code\&\#34;:200,\&\#34;data\&\#34;:\{\&\#34;name\&\#34;:\&\#34;张三\&\#34;\},\&\#34;msg\&\#34;:\&\#34;成功\&\#34;\}），PM能看懂“data里是返回的核心数据”即可。

PM重点：看接口文档时，能找到“请求参数”“响应数据”“状态码”，就能和研发确认“这个接口能满足需求吗”“返回的数据对不对”。

## 5\. Cookie / Session / Token（用户登录态，PM必懂权限逻辑）

核心问题：用户登录后，浏览器怎么“记住”用户？（总不能每次点击都重新登录）——这三个就是“记住登录态”的三种方式。

- Session：服务端记录——相当于“你去酒店住，前台给你一张房卡（SessionID），前台电脑里记着“房卡对应哪个房间”，你每次出示房卡，前台就知道你是谁。” 特点：数据存在服务器，安全，但服务器压力大（用户多了，要记很多Session）。

- Token / JWT：客户端携带——相当于“你去景区，买了一张门票（Token），门票上写着你的身份信息，你每次进景区，工作人员看门票就知道你是谁，不用查后台”。 特点：数据存在客户端（浏览器/APP），服务器压力小，现在最常用（比如微信小程序、APP登录，大多用JWT）。

- Cookie：客户端存储的小文件——可以存登录态，也可以存其他小数据（比如用户的偏好设置），但安全性低（容易被篡改），现在大多用来配合Token使用。

PM重点：知道“登录态是靠这三种方式实现的”，设计权限需求时（比如“未登录用户不能看个人中心”），能和研发说清“需要校验登录态”，并知道“Token过期后，用户需要重新登录”。

## 6\. 前后端分离（核心架构，PM必须懂）

通俗理解：以前的网站是“前端和后端绑在一起”（比如PHP网站），改一个按钮样式，都要动后端代码；现在的网站是“前端和后端分开”，各司其职，通过API沟通。

- 前端：负责“页面展示和用户交互”——比如按钮长什么样、点击按钮后页面怎么动、表单怎么填，不处理复杂逻辑。

- 后端：负责“业务逻辑和数据处理”——比如用户登录时校验密码、下单时扣减库存、查询用户数据，不关心页面长什么样。

- 沟通方式：前端通过“API接口”向后端发请求，后端处理后返回数据，前端再把数据展示出来。

PM重点：知道“前后端分离”的好处——前端和后端可以同时开发（比如前端做页面，后端做接口），效率高；改前端样式不用动后端，改后端逻辑不用动前端，维护方便。设计需求时，能区分“这个功能是前端做，还是后端做”。

# C\. 前端技术栈（PM视角，只懂“页面相关”，30分钟掌握）

## 1\. 前端是什么

一句话总结：**用户能看到、摸到、操作的所有部分**，包括：网页、APP界面、小程序、按钮、表单、图表、动画、弹窗，甚至是“点击按钮后的加载效果”“下拉刷新”。

核心目标：让用户“看得舒服、用得顺畅”，把后端返回的数据，变成用户能直观理解的界面。

补充：响应式页面——同一个页面，在电脑、手机、平板上展示的样式不一样（比如电脑上是两列，手机上是一列），目的是适配不同设备，PM设计需求时要考虑“响应式适配”。

## 2\. 前端三件套（不用懂代码，记作用）

- HTML：页面的“骨架”——相当于“盖房子的墙体”，定义页面有什么（比如有一个按钮、一个输入框、一个标题），只负责“有什么”，不负责“好不好看、能不能动”。

- CSS：页面的“样式和布局”——相当于“装修房子”，定义页面好不好看（比如按钮是红色的、字体是16号、标题居中）、布局怎么样（比如按钮在左边、输入框在右边）。

- JavaScript（JS）：页面的“交互和逻辑”——相当于“房子的水电”，让页面“动起来”（比如点击按钮后弹窗、输入内容后实时校验、下拉加载更多数据）。

PM重点：知道“页面的骨架、样式、交互，分别由这三个东西控制”，比如“改按钮颜色”是CSS的活，“点击按钮弹窗”是JS的活，“加一个输入框”是HTML的活，和研发对接时能精准描述需求。

## 3\. 前端框架（记名字、记作用，不用懂原理）

为什么要用框架？——如果开发复杂产品（比如微信小程序、淘宝网页），只用HTML\+CSS\+JS，代码会很乱、不好维护（比如一个按钮在多个页面用到，要写多遍代码），框架就是“帮前端省时间、规范代码”的工具。

最常用的两个框架（必记）：

- React：Facebook开发，适合复杂、大型产品（比如抖音网页版、知乎），特点是“组件化强、灵活”。

- Vue：国内最常用，适合中小型产品、小程序（比如微信小程序大多用Vue的衍生框架Uni\-app），特点是“简单、易上手”。

框架的核心好处（PM要懂）：

- 组件化：把页面拆成一个个“小零件”（比如按钮、卡片），重复使用，减少开发时间；

- 可维护：代码规范，后续改需求、修bug更方便；

- 适配复杂产品：比如产品有10个页面，每个页面都有相同的导航栏，用框架可以只写一次导航栏，所有页面复用，不用重复开发。

## 4\. 组件化思想（PM设计界面必懂）

通俗理解：组件 = 页面的“小零件”，一个页面 = 多个组件拼装而成，就像“搭乐高”——乐高零件（组件）可以重复用，拼不同的造型（页面）。

常见组件（PM能识别即可）：

- 基础组件：按钮、输入框、下拉框、单选框、复选框；

- 业务组件：用户卡片（显示用户头像、姓名）、订单卡片（显示订单号、金额）、弹窗（登录弹窗、提示弹窗）；

PM重点：设计界面时，要有“组件化思维”——比如“多个页面都需要用到用户卡片”，就可以让前端做一个“用户卡片组件”，重复使用，减少开发成本；同时要知道“组件复用性越高，开发越快”。

## 5\. 状态管理（PM理解“页面变化”的核心）

通俗理解：状态（state）= 页面的“当前状态”，比如“按钮是否被点击”“输入框里的内容”“列表是否加载完成”“用户是否登录”——页面的所有变化，本质上都是“状态的变化”。

- 举个例子：用户点击“登录”按钮，输入正确的账号密码，“登录状态”从“未登录”变成“已登录”，页面就会从“登录页”切换到“个人中心页”——这就是“状态变化触发页面更新”。

- 全局状态 vs 局部状态：
       

    - 全局状态：整个产品都能用的状态（比如用户登录状态、用户姓名），比如“用户登录后，所有页面都能显示用户姓名”；

    - 局部状态：只有某个页面能用的状态（比如某个输入框里的内容、某个弹窗是否显示），比如“个人中心的输入框内容，不会影响首页”。

PM重点：知道“用户的每一个操作，本质上都是在改变状态”，设计交互需求时，能说清“用户操作后，页面应该怎么变”（比如“点击提交按钮后，按钮变成加载状态，提交成功后显示提示弹窗”）。

## 6\. 常见前端配套（记名字、记作用，不用深入）

- Tailwind CSS：快速写CSS的工具，前端不用自己写复杂的样式代码，直接用现成的样式类（比如“红色按钮”“居中对齐”），节省时间——PM知道“用这个工具能加快前端开发速度”即可。

- Ant Design：现成的组件库，里面有各种做好的组件（按钮、表单、卡片），前端直接用，不用自己开发——PM知道“用组件库能减少开发时间，且界面风格统一”即可。

- 图表库（比如ECharts、AntV）：用来做数据可视化（比如折线图、柱状图、饼图）——PM设计“数据统计”需求时，知道“前端可以用这些库做图表”，不用自己从零开发。

- 路由系统：控制“页面跳转”，比如点击“首页”跳转到首页，点击“个人中心”跳转到个人中心——PM知道“路由就是页面跳转的规则”，设计页面跳转逻辑时，能和研发说清“从A页面点击按钮，跳转到B页面”。

## 7\. 产品经理要懂什么（核心重点，必记）

- 页面由组件构成：设计界面时，尽量复用组件，减少开发成本；

- 一个交互会触发状态变化：写需求时，明确“用户操作后，状态怎么变、页面怎么展示”；

- 哪些效果开发成本高：比如“复杂动画”“实时拖拽”“3D效果”，开发时间长，PM要评估“这个效果是否必要”；

- 什么是前端可实现/不易实现：
        

    - 易实现：改样式、简单交互（点击、输入）、页面跳转、简单数据展示；

    - 不易实现：复杂动画、实时协作（比如多人同时编辑一个文档）、大量数据的实时渲染（比如万条数据无卡顿展示）。

# D\. 后端技术栈（PM视角，只懂“逻辑和数据”，30分钟掌握）

## 1\. 后端是什么

一句话总结：**用户看不到的“背后逻辑”，负责处理前端的请求、操作数据、保障安全和权限**，相当于产品的“大脑”。

核心工作（必记）：

- 处理业务逻辑：比如用户登录时校验账号密码、下单时扣减库存、支付时对接支付接口、审核时判断是否通过；

- 提供API：给前端提供接口，让前端能获取数据、提交数据（比如前端要显示用户信息，就调用后端的“获取用户信息API”）；

- 操作数据库：把用户数据、订单数据、产品数据存到数据库，需要时再取出来；

- 处理权限、消息、任务：比如判断“这个用户能不能看这个页面”“给用户发邮件/短信”“处理耗时的任务（比如导出数据）”。

PM重点：知道“所有涉及‘数据存储、权限、复杂逻辑’的功能，都需要后端来做”，比如“用户登录校验”“订单支付”“数据统计”，不能让前端做（不安全、不可控）。

## 2\. 常见后端语言（记名字、记特点，不用学代码）

后端语言就是“写后端代码的工具”，不同语言有不同的特点，研发会根据产品需求选择，PM知道“不同语言的适用场景”即可：

- Python：简单、易上手，适合快速开发、AI相关产品（比如大模型调用、数据处理），现在AI产品后端大多用Python；

- JavaScript / Node\.js：和前端JS是同一种语言，适合前后端都用JS的产品（比如小程序、轻量级Web产品），开发效率高；

- Java：稳定、安全，适合大型企业级产品（比如银行、电商平台），承载高并发能力强，但开发速度相对慢；

- Go：速度快、并发能力强，适合高并发产品（比如直播、社交APP），现在越来越流行。

PM重点：不用纠结“选哪种语言”，但要知道“AI产品优先选Python”“高并发产品优先选Go/Java”，和研发讨论技术方案时能理解选择理由。

## 3\. 常见后端框架（记名字、记作用，不用懂原理）

后端框架和前端框架一样，是“帮后端省时间、规范代码”的工具，不同语言对应不同的框架：

- Python相关：FastAPI（快速开发API，适合AI产品）、Flask（轻量级，适合小型产品）；

- Node\.js相关：Express（轻量级）、NestJS（适合大型产品，规范）；

- Java相关：Spring Boot（最常用，适合企业级产品）。

PM重点：知道“用框架能加快后端开发速度”，比如“用FastAPI开发AI接口，比不用框架快很多”，评估开发周期时要考虑这一点。

## 4\. API 设计（PM看接口文档、提需求必懂）

API是“前后端沟通的桥梁”，API设计的好坏，直接影响开发效率和后续维护，PM要懂基本的API设计规范（不用自己设计，能看懂、能提意见即可）。

核心规范（必记）：

- RESTful API：最常用的API设计规范，简单说就是“用URL表示资源，用请求方式表示操作”，比如：
        

    - 获取用户信息：GET /api/user（URL是“用户资源”，请求方式是“获取”）；

    - 创建用户：POST /api/user（请求方式是“提交”）；

    - 修改用户：PUT /api/user/1（URL里的“1”是用户ID，请求方式是“修改”）；

    - 删除用户：DELETE /api/user/1。

- 路径设计：清晰、易懂，比如“/api/order”表示订单相关接口，“/api/user”表示用户相关接口，不要用复杂的路径（比如“/api/a1/b2/c3”）；

- 参数设计：明确“需要传什么参数”（比如登录接口需要传“用户名”“密码”）、“参数类型”（比如用户名是字符串、密码是字符串）；

- 返回格式：统一，比如所有接口都返回\{\&\#34;code\&\#34;:状态码,\&\#34;data\&\#34;:返回数据,\&\#34;msg\&\#34;:提示信息\}，方便前端统一处理；

- 错误码设计：明确不同错误的错误码，比如“用户名不存在”对应错误码1001，“密码错误”对应1002，方便排查问题。

PM重点：看接口文档时，能看懂“这个接口是做什么的”“需要传什么参数”“返回什么数据”，能和研发确认“这个接口能不能满足需求”。

## 5\. 业务逻辑（PM最熟悉，结合产品需求理解）

业务逻辑就是“产品的核心规则”，后端的核心工作就是“把产品规则翻译成技术代码”，PM作为需求提出者，最懂业务逻辑，只需知道“后端会如何处理这些逻辑”即可：

- 下单逻辑：用户点击“下单”→ 后端校验用户是否登录 → 校验库存是否充足 → 扣减库存 → 创建订单 → 返回下单结果；

- 登录逻辑：用户输入账号密码 → 后端校验账号密码是否正确 → 生成Token（登录态） → 返回Token和用户信息；

- 审核逻辑：用户提交审核申请 → 后端存储审核信息 → 通知审核人员 → 审核人员操作后 → 后端更新审核状态 → 通知用户；

- 支付逻辑：用户点击“支付”→ 后端对接支付平台（微信/支付宝） → 生成支付链接 → 用户支付后 → 支付平台通知后端 → 后端更新订单状态；

- 权限判断：用户访问某个页面/接口 → 后端校验用户角色 → 判断是否有权限访问 → 有权限则返回数据，无权限则返回401错误。

PM重点：写需求时，明确“业务逻辑的每一步”，比如“下单时必须校验库存，库存不足时提示‘库存不足’”，后端才能准确实现。

## 6\. 异步任务（PM理解“为什么有些功能不能实时”）

通俗理解：异步任务 = “后台默默做的任务，不影响用户当前操作”，比如“用户点击导出数据，不用一直等，导出完成后会收到通知”——如果用同步任务，用户需要一直等，页面会卡住，体验很差。

常见异步任务（必记）：

- 发邮件/短信：比如用户注册后发验证邮件、下单后发订单通知，不用用户等待；

- 文件处理：比如导出数据、上传大文件、视频转码，耗时较长，适合异步；

- AI长任务：比如大模型生成长文本、图片生成、语音转文字，耗时较长，不能让用户一直等；

为什么不能都同步做？——同步任务会“阻塞”，比如用户点击导出数据，如果用同步，页面会卡住，用户不能做其他操作，体验极差；异步任务则不会阻塞，用户可以继续操作其他功能，任务完成后再通知用户。

PM重点：设计需求时，判断“这个任务是不是耗时的”，如果是，就设计成“异步\+通知”的形式（比如“导出数据后，通过消息通知用户”），提升用户体验。

## 7\. 产品经理要懂什么（核心重点，必记）

- 接口文档怎么看：能找到“接口用途、请求参数、响应数据、状态码”，能和研发确认接口是否满足需求；

- 一个需求需要几个接口：比如“用户登录\+个人中心”，需要“登录接口”“获取用户信息接口”，可能还需要“修改用户信息接口”，能大概估算接口数量；

- 哪些逻辑适合放后端：涉及数据存储、权限、安全、复杂计算、耗时任务的逻辑，必须放后端；简单的展示逻辑、页面交互，可放前端；

- 为什么有些功能“不能实时”：因为是异步任务（耗时），比如“导出数据”“AI生成内容”，需要和用户说明“请耐心等待，完成后会通知你”。

# E\. 数据库与数据存储（PM懂“存数据”，25分钟掌握）

## 1\. 数据库是什么

通俗理解：数据库 = “产品的‘仓库’”，用来存储所有业务数据（用户信息、订单、对话记录、产品配置等），保障数据“持久化”（即使服务器重启，数据也不会丢）。

核心作用：

- 存储数据：把用户、订单等数据存起来，需要时再取；

- 支撑检索：比如“查询用户的所有订单”“搜索商品”，数据库能快速找到对应数据；

- 支撑统计：比如“统计今日下单量”“统计用户总数”，数据库能快速计算；

- 保障数据安全：比如防止数据丢失、被篡改。

PM重点：知道“所有需要长期保存的数据，都要存在数据库里”，比如用户登录信息、订单记录，不能存在前端（前端存的数据容易丢、容易被篡改）。

## 2\. 关系型数据库（最常用，必懂）

特点：数据是“结构化”的，像“Excel表格”一样，有行、列，不同表格之间有关联（比如“用户表”和“订单表”，通过“用户ID”关联），适合存储“结构化业务数据”（比如用户、订单、商品）。

最常用的两个（必记）：

- MySQL：最主流、最常用，适合中小型产品、大部分互联网产品（比如微信、淘宝的部分数据），开源、免费、稳定；

- PostgreSQL：功能更强大，适合复杂数据场景（比如地理信息、复杂查询），开源、免费。

核心概念（PM能看懂即可）：

- 表：相当于“Excel表格”，比如“用户表”“订单表”；

- 字段：相当于“Excel表格的列”，比如“用户表”有“用户ID、姓名、手机号、密码”等字段；

- 记录：相当于“Excel表格的行”，比如“用户表”里的一行数据，就是一个用户的信息；

- 主键（ID）：每个表的“唯一标识”，比如“用户ID”，每个用户的ID都不一样，用来区分不同用户；

- 外键：用来关联两个表，比如“订单表”里的“用户ID”，就是外键，关联“用户表”的“用户ID”，表示“这个订单属于哪个用户”。

PM重点：设计数据需求时，要有“表结构思维”，比如“用户有哪些信息”“订单和用户是什么关系”，能和研发讨论“表应该怎么设计”。

## 3\. 非关系型数据库（NoSQL，记作用、记场景）

特点：数据是“非结构化”的，不像Excel表格，没有固定的列，适合存储“灵活、非结构化的数据”（比如用户的聊天记录、朋友圈内容），或者“需要高速访问的数据”。

最常用的两个（必记）：

- MongoDB：文档型数据库，数据像“JSON”格式（key\-value），适合存储非结构化数据（比如用户的聊天记录、文章内容、图片描述）；

- Redis：缓存数据库，速度极快，适合存储“高频访问、不常修改”的数据（比如热门商品、验证码、用户登录态）——后面单独讲。

PM重点：知道“非结构化数据（比如聊天记录）用MongoDB，高频访问数据用Redis”，和研发讨论数据存储方案时能理解选择理由。

## 4\. Redis / 缓存（PM必懂，高频用到）

通俗理解：Redis = “数据库的‘高速缓存区’”，相当于“你把常用的东西放在桌子上，不用每次都去柜子里拿”——把高频访问、不常修改的数据，存到Redis里，浏览器请求时，先从Redis里取，取不到再去数据库里取，加快访问速度。

核心作用：

- 加快访问速度：Redis访问速度是MySQL的10倍以上，比如“热门商品列表”，用户每次访问都从Redis取，不用去数据库，页面加载更快；

- 存热点数据：比如热门商品、热门文章、验证码（有效期5分钟）、用户登录态（Token）；

- 减少数据库压力：如果所有请求都去访问数据库，数据库压力会很大，容易崩；用Redis缓存后，大部分请求会从Redis取，减少数据库的访问量。

PM重点：设计需求时，判断“哪些数据是高频访问、不常修改的”，比如“首页热门内容”“验证码”，可以让研发用Redis缓存，提升页面加载速度；同时知道“Redis里的数据可能会过期”（比如验证码有效期5分钟），设计交互时要考虑这一点。

## 5\. 文件存储 / 对象存储（PM懂“存文件”即可）

通俗理解：文件存储 = “专门存‘大文件’的仓库”，比如图片、视频、音频、文档（PDF、Word）——这些文件不能存在数据库里（数据库适合存小的结构化数据），需要专门的文件存储工具。

常见场景：

- 图片：用户头像、商品图片、文章封面；

- 视频：用户上传的视频、产品宣传视频；

- 音频：语音消息、音乐；

- 文档：用户上传的PDF、Word、Excel。

常用工具（记名字即可）：阿里云OSS、腾讯云COS、AWS S3——都是云服务提供的文件存储工具，研发会直接用。

PM重点：知道“大文件要存在文件存储里，不能存在数据库里”，设计“文件上传”需求时，要考虑“文件大小限制”（比如“最多上传100M的视频”）、“文件格式限制”（比如“只能上传图片格式”）。

## 6\. 向量数据库（AI产品PM必懂）

通俗理解：向量数据库 = “专门存‘向量’的数据库”，向量是“把文本、图片等转换成的一串数字”（比如“你好”转换成\[0\.1, 0\.2, 0\.3\.\.\.\]），核心作用是“做相似度搜索”——这是AI产品（尤其是RAG知识库）的核心。

核心场景（必记）：

- AI检索（RAG）：比如“用户提问‘什么是RAG’，系统从知识库中找到和‘什么是RAG’最相似的片段，再给大模型回答”——这个“找相似片段”的过程，就是向量数据库完成的；

- 存embedding：embedding就是“文本/图片转换成的向量”，向量数据库专门用来存这些向量，方便快速检索。

常见向量数据库（记名字即可）：Chroma（轻量级，适合开发测试）、Pinecone（云服务，适合生产环境）、Elasticsearch（既能做全文检索，也能做向量检索）。

PM重点：知道“AI检索（RAG）必须用向量数据库”，设计RAG需求时，能和研发说清“需要把知识库文档转换成向量，存到向量数据库里，用于相似度搜索”。

## 7\. 产品经理要懂什么（核心重点，必记）

- 什么数据存在数据库：结构化数据（用户、订单、配置），需要长期保存、需要检索/统计的数据；

- 什么数据存在文件存储：大文件（图片、视频、文档），不能存在数据库里的数据；

- 为什么要缓存：高频访问、不常修改的数据，用缓存加快访问速度、减少数据库压力；

- 为什么AI检索要向量库：向量库能快速做“相似度搜索”，找到和用户提问最相关的知识片段，这是RAG的核心，普通数据库做不到。

# F\. AI 技术栈（AI产品PM核心，40分钟掌握，重点中的重点）

## 1\. 模型层（记类型、记作用，不用懂训练）

核心：AI产品的“智能核心”，所有AI能力（文本生成、图片生成、语音识别）都来自模型，PM要懂“不同模型的适用场景”，知道“什么时候用什么模型”。

常见模型类型（必记）：

- LLM 大语言模型：处理文本的核心模型，比如GPT\-4、文心一言、通义千问，能做文本生成（写文章、写文案）、文本分类（给评论打标签）、文本抽取（提取关键词）、问答（回答用户问题）；

- 文本模型：专门处理文本的模型，比如BERT（文本分类、抽取）、RoBERTa（优化版BERT），适合简单的文本任务；

- 图像模型：处理图片的模型，比如MidJourney、Stable Diffusion（图片生成）、YOLO（目标检测，比如识别图片里的人、物）；

- 语音模型：处理语音的模型，分为ASR（语音转文字，比如微信语音转文字）、TTS（文字转语音，比如导航语音）；

- 多模态模型：能同时处理文本、图片、语音的模型，比如GPT\-4V、文心一格，能“看图说话”“语音提问\+图片回答”。

PM重点：知道“不同AI需求对应不同模型”，比如“写文案用LLM”“生成图片用图像模型”“语音转文字用ASR模型”，设计AI需求时能明确“需要什么类型的模型”。

## 2\. Prompt 工程（AI产品PM必懂，直接影响AI效果）

通俗理解：Prompt = “给AI的‘指令’”，比如“写一篇关于春天的文案，风格活泼，100字以内”——这个指令就是Prompt，Prompt的好坏，直接决定AI的输出效果。

核心组成（必记，写Prompt时会用到）：

- System Prompt（系统提示）：给AI设定“角色和规则”，比如“你是一个专业的文案写手，只写活泼风格的文案，不写严肃内容”，作用是“约束AI的输出风格和范围”；

- User Prompt（用户提示）：用户的核心需求，比如“写一篇关于春天的文案”；

- 输出格式约束：明确AI的输出格式，比如“输出格式为：标题\+正文，正文100字以内”，避免AI输出杂乱无章；

- 示例驱动（Few\-shot）：给AI举例子，比如“比如：‘春天来了，微风拂面，桃花开了，一切都刚刚好～’，按照这个风格写”，让AI更懂你的需求；

- 提示词调优：优化Prompt，让AI输出更精准，比如把“写一篇春天的文案”优化成“写一篇关于春天的朋友圈文案，风格活泼，带emoji，100字以内，突出桃花、微风两个元素”。

PM重点：知道“Prompt能影响AI输出效果”，设计AI交互时，要考虑“给用户提供Prompt模板”（比如让用户选择“文案风格”“输出长度”），或者“后台预设System Prompt”，确保AI输出符合产品需求。

学习补充（10分钟看懂）：[OpenAI Prompt 工程指南（中文版，只看基础部分）](https://platform.openai.com/docs/guides/prompt-engineering)

## 3\. Inference 推理（AI产品PM懂“响应逻辑”即可）

通俗理解：Inference（推理）= “AI处理用户输入，生成输出的过程”，比如用户输入“写一篇春天的文案”（输入），AI处理后生成文案（输出）——这个过程就是推理。

核心流程：用户输入 → 前端把输入传给后端 → 后端把输入传给模型 → 模型推理处理 → 模型返回结果 → 后端处理结果（比如过滤敏感内容） → 前端展示给用户。

PM重点关注3个点（影响用户体验和成本）：

- 延迟：推理的速度，比如“AI生成文案需要1秒”还是“需要5秒”——延迟越高，用户体验越差，PM要和研发确认“延迟控制在多少以内”；

- 成本：推理需要消耗算力，算力是要花钱的——比如“生成1000字文案，成本0\.1元”，PM要考虑“成本和体验的平衡”（比如不需要追求极致速度，可降低算力，减少成本）；

- 稳定性：推理过程是否稳定，会不会经常报错、生成失败——PM要要求研发“做好异常处理”，比如推理失败时，提示用户“请重试”。

## 4\. Embedding（AI检索核心，必懂）

通俗理解：Embedding = “把文本、图片等‘转换成一串数字’”，比如“你好”转换成\[0\.1, 0\.2, 0\.3, \.\.\.\]，这串数字就是Embedding（向量）——核心作用是“表示语义”，语义越相似，Embedding的数字越接近。

核心用途：

- 文本向量化：把知识库文档、用户提问，都转换成Embedding；

- 相似度计算：通过比较两个Embedding的“距离”，判断语义相似度——比如用户提问“什么是RAG”的Embedding，和知识库中“RAG是检索增强生成”的Embedding距离很近，就认为这两个文本语义相似；

- 语义搜索基础：RAG的核心就是“通过Embedding相似度，找到和用户提问最相关的知识片段”，没有Embedding，就没有RAG。

PM重点：知道“Embedding是RAG的基础”，设计RAG需求时，要明确“需要把知识库文档转换成Embedding，存到向量数据库里”，同时知道“Embedding模型的选择，会影响检索效果”（比如用更好的Embedding模型，相似度计算更精准）。

## 5\. RAG（AI产品PM最常用，必懂原理和场景）

RAG全称：检索增强生成（Retrieval\-Augmented Generation），通俗理解：**让大模型“先查资料，再回答”**——解决大模型“知识过时、容易说胡话（幻觉）”的问题。

核心流程（必记，能给研发讲清楚）：

1. 准备知识库：先整理产品所需的知识素材（比如产品手册、行业文档、常见问题等），对素材进行拆分（拆成短片段，方便后续检索），避免大段文本导致检索不精准；

2. 文本向量化：通过Embedding模型，将拆分后的知识库片段、用户的提问，都转换成向量（Embedding），相当于给每一段文本“贴一个数字标签”；

3. 向量存储：把转换好的知识库向量，存入向量数据库（比如Chroma、Pinecone），向量数据库会优化存储和检索逻辑，确保后续能快速找到相似向量；

4. 用户提问与检索：用户输入提问后，系统先将提问转换成向量，再到向量数据库中，搜索与该向量“距离最近”（语义最相似）的3\-5个知识库片段（避免单一片段遗漏信息）；

5. Prompt拼接：将检索到的相似知识库片段，和用户的提问、预设的System Prompt，一起拼接成一个完整的Prompt，传给大语言模型；

6. 模型生成与反馈：大模型结合拼接的Prompt（相当于“带着资料回答”），生成精准、无幻觉的回答，再通过前端展示给用户；若检索不到相关片段，会提示用户“未找到相关知识”，避免胡编乱造。

补充说明（PM重点）：RAG的核心价值的是“让AI回答有依据、不过时”，比如产品更新了功能手册，只需更新知识库，不用重新训练大模型，降低成本；PM设计RAG需求时，要明确“知识库的范围”“检索的精准度要求”（比如只检索近3个月的文档），还要考虑“知识库的更新频率”（比如每周更新1次产品手册）。

常见应用场景（PM必记）：

- 智能客服：用户提问“如何修改密码”，系统检索知识库中“密码修改流程”片段，大模型结合片段生成步骤化回答；

- 产品帮助中心：用户查询“订单退款规则”，系统快速检索相关规则，生成精准回复，不用用户手动翻找长文档；

- 内部知识库检索：员工提问“项目接口规范”，系统检索内部文档，生成符合规范的回答，提升工作效率；

- AI问答产品：比如行业问答工具，用户提问“行业最新政策”，系统检索最新政策文档，生成有依据的回答，避免知识过时。

PM避坑点：① 知识库拆分不能太粗（比如整本书拆成1段），否则检索精准度低；也不能太细（比如每句话拆1段），否则检索效率低；② 要设置“检索阈值”，比如只有相似度达到80%以上的片段，才拼接给大模型，避免无关片段干扰回答；③ 定期清理知识库中的过期内容（比如旧版产品手册），避免AI回答过时信息。
